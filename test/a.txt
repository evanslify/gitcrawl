&lt;h2>&lt;a id="user-content-the-idea" class="anchor" href="#the-idea" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>The Idea&lt;/h2>

&lt;p>Most virtual reality experiences today are passive ones. We've all enjoyed Rift Coaster and the like, and these are promising first steps. But the promise of VR is &lt;em>reality&lt;/em>… immersion. &lt;/p>

&lt;p>Or project is another step in the direction of immersion. We largely have two components to our product - the Nerf gun and gesture controls. Both of these components are fitted with enough sensors to let you forget that you're using a controller and let you focus instead on surviving in the zombie infested world of Left 4 Dead 2.&lt;/p>

&lt;p>&lt;a href="https://camo.githubusercontent.com/77406326f7f36af3517a87ae366772be1dd0080e/687474703a2f2f73332e616d617a6f6e6177732e636f6d2f6368616c6c656e6765706f73742f70686f746f732f70726f64756374696f6e2f736f6c7574696f6e5f70686f746f732f3030302f3230302f3231392f64617461732f786c617267652e706e673f31343231353931313431" target="_blank">&lt;img src="https://camo.githubusercontent.com/77406326f7f36af3517a87ae366772be1dd0080e/687474703a2f2f73332e616d617a6f6e6177732e636f6d2f6368616c6c656e6765706f73742f70686f746f732f70726f64756374696f6e2f736f6c7574696f6e5f70686f746f732f3030302f3230302f3231392f64617461732f786c617267652e706e673f31343231353931313431" alt="Final Hackathon Product" data-canonical-src="http://s3.amazonaws.com/challengepost/photos/production/solution_photos/000/200/219/datas/xlarge.png?1421591141" style="max-width:100%;">&lt;/a>
&lt;a href="https://raw.githubusercontent.com/wchill/Left4VR/master/pics/EOH_2015.jpg" target="_blank">&lt;img src="https://raw.githubusercontent.com/wchill/Left4VR/master/pics/EOH_2015.jpg" alt="Version 2 at EOH" style="max-width:100%;">&lt;/a>&lt;/p>

&lt;p>This project is built with a special focus on Left 4 Dead 2 as we understand that a good virtual reality experience requires accuracy and finesse just as much as it requires general functionality. In this regard, while our project will work with most first person shooter games with some tweaking, we have intentionally decided to showcase it using Left 4 Dead 2 only. Left 4 Dead 2 was chosen because of its emphasis on cooperative play, meaning that one does not need to worry about trying to compete against very skilled keyboard + mouse users. It also provides a lot of room for using gestures in a comfortable, natural way. We are currently looking into porting this system to the Borderlands and Metro series.&lt;/p>

&lt;h2>&lt;a id="user-content-design" class="anchor" href="#design" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Design&lt;/h2>

&lt;h3>&lt;a id="user-content-armed-controller" class="anchor" href="#armed-controller" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Armed Controller&lt;/h3>

&lt;p>The dominant controller for the project is the Armed Controller. It looks like a gun and use for this controller is clear enough: aiming down sights, firing, reloading, etc. It started off as an ordinary Nerf Rapidstrike, which we then gutted to make space for sensors and sensor feeds. We chose this model of the Nerf for its form factor (which is similar to most guns in fps games), its versatility (the collapsible stock, design and multiple tactical rails) and for its electrically controlled trigger system (most moving parts were electrically driven, not mechanically). All the preexisting switches from the Rapidstrike feed into a Raspberry Pi which is mounted on the side of the gun. There is also a switch that detects whether a magazine is inserted into the controller and a reload action is performed accordingly. &lt;/p>

&lt;p>We’ve also removed the contents of what used to be battery compartment of the Rapidstrike and fitted a Nintendo Wii Remote, MotionPlus and Nunchuck controller there. This controller is used to get information about the Armed Controller’s movements and to control movement within the game (walking, sprinting, control of an under-barrel attachment such as a flashlight, etc). These communicate directly with the PC via Bluetooth. The plan is to remove the Wii Remote completely from the equation and have the MotionPlus and Nunchuk interface with the Pi directly over I2C to reduce software complexity, power consumption and response time.&lt;/p>

&lt;p>Power is supplied to the Armed Controller by the mounted Raspberry Pi, which in turn can be powered via Micro USB. We could therefore power this part of the project using a battery pack/wall adapter/similar - currently an off-the-shelf USB battery pack is used. The Raspberry Pi uses Wi-Fi to communicate with the gaming computer, with real-world latency around 2ms, but Ethernet is also an option if even lower latency is desired/needed. We have support for Bluetooth as well, although the response times using the Pi's USB stack aren’t desirable (in the order of tens of milliseconds depending on the environment) and in an application like VR where fast response times are crucial, we found this to be an unacceptable compromise for our needs.&lt;/p>

&lt;h3>&lt;a id="user-content-gesture-controls" class="anchor" href="#gesture-controls" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Gesture Controls&lt;/h3>

&lt;p>Another major component of the build is the gesture controls. Currently this consists of a custom-made glove worn on the left hand and a Myo gesture control armband on the right forearm. This is to be used to perform most other gestures in the game. This includes (but not limited to) picking up items, using throwables, signaling fellow players, and interacting with fellow players and the environment.&lt;/p>

&lt;p>There are two flex sensors in the glove that we use to read the state of the hand. These sensors are sampled at 100Hz using a Spark Core, which feeds data via Wi-Fi to the PC. The Myo actually commmunicates with another computer using Bluetooth, then the data is processed using the myo-python library and transmitted over the network back to the PC. The plan is to remove this component and replace it with the IMUs in the Nunchuk/MotionPlus on the Armed Controller.&lt;/p>

&lt;p>Currently we plan on improving the glove such that it has 5 flex sensors (one for each finger) and also conductive pads to allow extra gestures by touching fingertips together.&lt;/p>

&lt;h3>&lt;a id="user-content-the-oculus" class="anchor" href="#the-oculus" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>The Oculus&lt;/h3>

&lt;p>The Oculus Rift DK2 completes our set up by providing the visuals to the user as well as the ability to look around the world using its excellent head tracking. In the initial version the Vireio Perception driver was used; however currently VorpX is being used.&lt;/p>

&lt;h3>&lt;a id="user-content-software-stack" class="anchor" href="#software-stack" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Software Stack&lt;/h3>

&lt;p>The software stack has undergone several iterations to improve code reusability and modularity. The goal is to make it simple to add any additional peripheral devices, supported games or keybindings that we want to use.&lt;/p>

&lt;p>Currently much of the processing is done on the PC itself using a Python script. Using the Twisted async networking library, it listens on a socket for incoming connections and converts the incoming data into virtual input. Currently all actions are hardcoded, but code is currently being developed in order to modularize this so that event triggers/actions can be dynamically added by peripheral devices. There is no security; this may or may not be fixed in the future. Any peripheral devices that wish to control the host computer can simply open up a socket connection to the given address and port. This means that if one peripheral fails (due to power, mechanical failure, or some other reason), the rest of the system will remain running. This is a huge improvement over how the system was implemeneted at MHacks V, where everything had to be powered on in a certain order.&lt;/p>

&lt;p>The Wii Remote/Nunchuk/MotionPlus, while they communicate via Bluetooth to the PC, do not directly control the script. They pass through another software layer - FreePIE, an input emulator which provides Python scripting functions to make use of popular controllers. Our FreePIE script also opens up a socket to the master script. If the gaming computer does not have Bluetooth, another computer like a laptop can instead run FreePIE, allowing some flexibility.&lt;/p>

&lt;p>The configuration of the software stack also allows for other features beyond this project; for example, it would be possible to write a mobile app to turn a smartphone or tablet into a very capable gaming controller, or to use another computer's keyboard and mouse for remote control/gaming.&lt;/p>

&lt;h2>&lt;a id="user-content-limitations" class="anchor" href="#limitations" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Limitations&lt;/h2>

&lt;p>We came close to the product we had originally envisioned, however there were a number of items that we did not have time to implement or are not currently possible.&lt;/p>

&lt;ul>&lt;li>&lt;strong>Microsoft Kinect&lt;/strong>: We wanted to use the Microsoft Kinect to watch the body’s posture to adjust for crouching, running, aiming down sights and the like. Unfortunately we were not able to do this because of the time restriction so we improvised and used the Nintendo Nunchuk Controllers instead.&lt;br>&lt;/li>
&lt;li>&lt;strong>Haptic feedback&lt;/strong>: Adding in vibration motors into the Nerf gun would allow for more realistic feeling controls.&lt;br>&lt;/li>
&lt;li>&lt;strong>Reload enforcement&lt;/strong>: In most games guns are reloaded automatically when the magazine is empty. Obviously we would like to disable this for greater realism. However currently this is not possible in most games. It can be accomplished in Left 4 Dead 2 using a server-side trick, but the animations will not look correct and it only works with Source-based multiplayer games.&lt;/li>
&lt;/ul>&lt;h2>&lt;a id="user-content-problems-encountered" class="anchor" href="#problems-encountered" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Problems encountered&lt;/h2>

&lt;h4>&lt;a id="user-content-initial-version" class="anchor" href="#initial-version" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Initial version&lt;/h4>

&lt;p>Because of the lack of native VR support in Left 4 Dead, we had to use a 3rd party driver called &lt;a href="http://www.mtbs3d.com/index.php?option=com_content&amp;amp;view=category&amp;amp;id=169&amp;amp;Itemid=490">Vireio Perception&lt;/a>. This caused multiple issues:  &lt;/p>

&lt;ul>&lt;li>&lt;del>The Myo Control software would conflict with Perception as Perception was attempting to inject its own graphical DLLs into Myo Control, causing errors. This required launching our setup in a particular order (run Perception with admin privileges -&amp;gt; launch L4D -&amp;gt; close Perception -&amp;gt; launch Myo Control)&lt;/del> Using myo-python fixes this problem; in addition changes are being made to use the MotionPlus instead of the Myo.&lt;/li>
&lt;li>&lt;del>Switching screens caused Left 4 Dead to crash on startup, complaining about "Failed to create D3D device!" Because this happened on-stage right before we were to present, we were unable to fix this issue in time. Cause (and solution) remains unknown.&lt;/del> No longer seems to be an issue on either Vireio or VorpX, most likely driver-related.&lt;br>&lt;/li>
&lt;li>Mirroring the display on the Rift was nontrivial. &lt;del>We had to use &lt;a href="https://obsproject.com/">Open Broadcaster Software&lt;/a>, a tool that streamers normally use, to capture the video from Left 4 Dead so it could be displayed on an external monitor. Unfortunately, because of how Perception works, we didn't get a nice stereo view like you would expect, but instead we got one that seemed very jittery (likely because it was switching between left and right views quickly).&lt;/del> OBS no longer works. We now use a modified game server with a spectator plugin and another computer with another copy of Left 4 Dead 2 to do a visual preview. It does not show the HUD or any Rift-specific distortions and is not quite real-time, but it is better than using OBS.&lt;br>&lt;/li>
&lt;li>The Oculus Rift had to be run in extended display mode as the primary monitor, making setting everything up an immense hassle because all our windows would show up on the Rift and there would be nothing on the desktop monitor.&lt;/li>
&lt;/ul>&lt;h4>&lt;a id="user-content-current-version" class="anchor" href="#current-version" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Current version&lt;/h4>

&lt;p>We have now switched to VorpX after some improvements to our stack, which has its own problems unfortunately.&lt;/p>

&lt;ul>&lt;li>&lt;del>When an item pickup is visible, the level geometry may flash white.&lt;/del> No longer an issue in the latest version of VorpX&lt;/li>
&lt;li>Open Broadcaster Software no longer works using either Vireio or VorpX. For spectating, another machine is used along with a L4D2 server that allows spectating in coop mode. Obviously this is not a portable solution, but for Source-based games it is sufficient.&lt;/li>

&lt;/ul>&lt;h2>&lt;a id="user-content-features" class="anchor" href="#features" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Features&lt;/h2>

&lt;ul>&lt;li>Moves you to the lane most likely to give you gold, prioritized like so:

&lt;ol>&lt;li>The lane with a Treasure Minion or Boss&lt;/li>
&lt;li>The lane with the Miniboss with the lowest health&lt;/li>
&lt;li>The lane with a Spawner below 40% health&lt;/li>
&lt;li>The lane with a Creep below 10% health&lt;/li>
&lt;li>The lane with the Spawner with the lowest health&lt;/li>
&lt;/ol>&lt;/li>
&lt;li>Activates most reusable abilities, if they are purchased and cooled down:

&lt;ul>&lt;li>Medics if your health is below 50%&lt;/li>
&lt;li>Morale Booster, Napalm, and Cluster Bombs if the lane has a Spawner and 2-3 Creeps&lt;/li>
&lt;li>Good Luck Charm as soon as possible&lt;/li>
&lt;li>Tactical Nuke if the current Spawner is between 60% and 30% health&lt;/li>
&lt;/ul>&lt;/li>
&lt;li>Activates some items if you have them and the situation calls for them:

&lt;ul>&lt;li>God Mode if Medics is in cooldown and your health is low&lt;/li>
&lt;li>Cripple Spawner if the spawner in the current lane has more than 95% health&lt;/li>
&lt;li>Gold Rain if facing a Boss who has more than 60% health&lt;/li>
&lt;/ul>&lt;/li>

&lt;ul>&lt;li>Moves you to the lane most likely to give you gold, prioritized like so:

&lt;ol>&lt;li>The lane with a Treasure Minion or Boss&lt;/li>

&lt;li>The lane with a Spawner below 40% health&lt;/li>
&lt;li>The lane with a Creep below 10% health&lt;/li>
&lt;li>The lane with the Spawner with the lowest health&lt;/li>
&lt;/ol>&lt;/li>
&lt;li>Activates most reusable abilities, if they are purchased and cooled down:

&lt;li>Tactical Nuke if the current Spawner is between 60% and 30% health&lt;/li>
&lt;/ul>&lt;/li>

&lt;li>Gold Rain if facing a Boss who has more than 60% health&lt;/li>
&lt;/ul>&lt;/li>

&lt;li>Disables certain abilities and items if facing a Boss (to try to maximize Raining Gold and Metal Detector benefits)&lt;/li>

&lt;p>(If you got the source code and want to compile it, see &lt;code>modifying_en.txt&lt;/code>,
 included on GitHub as &lt;code>doc/dstwo/modifying_en.txt&lt;/code>.)&lt;/p>

&lt;h2>&lt;a id="user-content-the-gba-bios" class="anchor" href="#the-gba-bios" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>The GBA BIOS&lt;/h2>

&lt;p>As of TempGBA version 1.45, an open-source BIOS replacement based on VBA-M's
high-level BIOS emulation is bundled in the release archive. This BIOS is
provided by Normmatt.&lt;/p>

&lt;p>For better compatibility with certain games, you may instead use the real
GBA BIOS, if you have it.&lt;/p>

&lt;p>The file you dump or find should be 16 KiB (16384 bytes) and have the following
checksums:&lt;/p>

&lt;ul>&lt;li>CRC32: &lt;code>81977335&lt;/code>;&lt;/li>
&lt;li>MD5: &lt;code>a860e8c0b6d573d191e4ec7db1b1e4f6&lt;/code>;&lt;/li>
&lt;li>SHA-1: &lt;code>300c20df6731a33952ded8c436f7f186d25d3492&lt;/code>.&lt;/li>
&lt;/ul>&lt;p>Once dumped or found, name the file &lt;code>gba_bios.bin&lt;/code>, and replace it in the
&lt;code>TEMPGBA&lt;/code> directory of your storage card. So the path should look like this:
&lt;code>/media/Your-Card/TEMPGBA/gba_bios.bin&lt;/code> on Linux; &lt;code>E:\TempGBA\gba_bios.bin&lt;/code>
on Windows.&lt;/p>

&lt;h1>&lt;a id="user-content-improving-game-compatibility-with-game_configtxt" class="anchor" href="#improving-game-compatibility-with-game_configtxt" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Improving game compatibility with game_config.txt&lt;/h1>

&lt;p>Included with this release is a file called game_config.txt. The file is the
same as the one included in gpSP Kai. It goes in the &lt;code>TEMPGBA&lt;/code> directory on
your storage card when you extract the release.&lt;/p>

&lt;p>This file contains information to enhance game compatibility in two ways.&lt;/p>

&lt;ul>&lt;li>Firstly, it tells TempGBA about the save type used by a certain game, so that
battery-backed saves may be read and written correctly.&lt;/li>
&lt;li>Secondly, it tells TempGBA about idle loops in a certain game, so that it may
stop emulating when the game reaches them. It runs the game faster.&lt;/li>
&lt;/ul>&lt;p>game_config.txt receives updates from the community occasionally.
For the latest version, see:
&lt;a href="http://filetrip.net/nds-downloads/flashcart-files/latest-x-f12630.html">http://filetrip.net/nds-downloads/flashcart-files/latest-x-f12630.html&lt;/a>&lt;/p>

&lt;h1>&lt;a id="user-content-using-ndsgba-saved-states" class="anchor" href="#using-ndsgba-saved-states" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Using NDSGBA saved states&lt;/h1>

&lt;p>The NDSGBA saved state format was not suitable for modifying the emulator,
because each vew version meant that older saved states could corrupt the new
version's memory.&lt;/p>

&lt;p>To play a NDSGBA saved state, you may do the following:&lt;/p>

&lt;ol>&lt;li>Load the version of NDSGBA you used to create your saved state.&lt;/li>
&lt;li>Load the game, then the saved state. Save in-game, then exit NDSGBA.
(Exiting NDSGBA is what creates the .sav file. TempGBA would not need
to exit.)&lt;/li>
&lt;li>Using your computer, copy the .sav file from the /NDSGBA/GAMERTS directory
on your storage card into /TEMPGBA/SAVES. You can do this step for multiple
games at once if you repeated steps 1 and 2 for them.&lt;/li>
&lt;li>Load TempGBA.&lt;/li>
&lt;li>If you want to create a saved state (a .sav file works too), load the game,
which will load the .sav file, and then create a TempGBA saved state.&lt;/li>
&lt;/ol>&lt;h1>&lt;a id="user-content-game-compatibility" class="anchor" href="#game-compatibility" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Game compatibility&lt;/h1>

&lt;p>The GBATemp community maintains a compatibility list for TempGBA. You can
see and edit it here: &lt;a href="http://wiki.gbatemp.net/wiki/TempGBA_Compatibility">http://wiki.gbatemp.net/wiki/TempGBA_Compatibility&lt;/a>&lt;/p>

&lt;h1>&lt;a id="user-content-rewinding" class="anchor" href="#rewinding" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Rewinding&lt;/h1>

&lt;p>You can rewind the game up to 10 steps of a user-defined duration, ranging from
a quarter of second to 10 seconds. You are granted 100 seconds of rewinding by
default, in steps of 10 seconds, using the button combination L+Y. This allows
you to quickly back out of a dangerous situation in a game.&lt;/p>

&lt;p>You can adjust the step time for rewinding in the Tools menu.&lt;/p>

&lt;p>You can adjust the hotkey used for rewinding in the Tools menu's hotkey
submenus. See the &lt;code>Hotkeys&lt;/code> section below for more information about these.&lt;/p>

&lt;h1>&lt;a id="user-content-frame-skipping" class="anchor" href="#frame-skipping" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Frame skipping&lt;/h1>

&lt;p>In the Video &amp;amp; audio menu, the &lt;strong>Frame skipping&lt;/strong> option allows you to select
a number of frames to skip between rendered frames.&lt;/p>

&lt;p>For some games, you may need to adjust frame skipping.&lt;/p>

&lt;ul>&lt;li>If a game runs at about 10 frames per second, setting frame skipping
to 1 will allow you to jump, move or shoot at the right times.&lt;/li>
&lt;li>If you want to show more frames per second in a game that already shows 20,
setting frame skipping to 1 or 0 will cause more frames to appear.
If using frame skipping 0 and the audio crackles too much, the current game
may crash the Supercard DSTwo itself, and may require rebooting your Nintendo
DS.&lt;/li>
&lt;li>Setting this to 10 will skip 10 frames and render one. You will find yourself
unable to perform actions during the correct frame with the DS buttons. It is
advised to set frame skipping to the lowest value with which you can play a
game.&lt;/li>
&lt;/ul>&lt;h1>&lt;a id="user-content-hotkeys" class="anchor" href="#hotkeys" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Hotkeys&lt;/h1>

&lt;p>&lt;strong>Important: Hotkeys are resolved as DS keys, not remapped GBA keys.&lt;/strong>
You can set hotkeys to use buttons that are left over from button remapping if
you don't use the rapid fire A and B buttons.&lt;/p>

&lt;p>You can set buttons to press to perform certain actions. For each action,
there is a &lt;em>global hotkey&lt;/em> and a &lt;em>game-specific override hotkey&lt;/em>. You might,
for example, want to have the DS R button bound to Rewind, but a specific game
uses a button mapped to it (see the section &lt;code>Button mapping&lt;/code> below for more
information) for something important. In that case, you can set the
global hotkey to R and make an override with X for that game.&lt;/p>

&lt;p>Hotkeys are sent to the current game, after being remapped, as well as
to their corresponding action. The criterion for a hotkey is met when
&lt;strong>at least&lt;/strong> all of its buttons are held. Additional keys are sent to the game
and can trigger another hotkey. For example, setting a hotkey to L and another
to R+X, then pressing L+R+X+Y will trigger both and send all GBA buttons mapped
to L, R, X and Y to the game.&lt;/p>

&lt;p>Available actions are:&lt;/p>

&lt;ul>&lt;li>Go to main menu. In addition to tapping the Touch Screen to return to
the main menu, you can set a hotkey to do the same.&lt;/li>
&lt;li>Temporary fast-forward. While this hotkey is held, the fast-forward option
will be forced on.&lt;/li>
&lt;li>Rewind. A hotkey needs to be set here if you want to use the rewind feature
in a game. After each second the hotkey is held, the game will rewind one
step. See the &lt;code>Rewinding&lt;/code> section above for more information.&lt;/li>
&lt;li>Toggle sound. Each time this hotkey is held, the sound will be disabled if
it's currently enabled, and vice-versa.&lt;/li>
&lt;li>Save state #1. Each time this hotkey is held, saved state #1 will be written,
without confirmation if it exists.&lt;/li>
&lt;li>Load state #1. Each time this hotkey is held, saved state #1 will be loaded.&lt;/li>
&lt;/ul&gt;&lt;h1>&lt;a id="user-content-button-mapping" class="anchor" href="#button-mapping" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Button mapping&lt;/h1>

&lt;p>A Nintendo DS has more buttons than a Game Boy Advance. This emulator allows
you to remap your GBA buttons to different DS buttons as well as use rapid fire
A and B buttons that trigger at 30 Hz (held for one frame, released the next).&lt;/p>

&lt;p>You can find the button mapping configuration in the Tools menu. For each
mapping, there is a &lt;em>global mapping&lt;/em> and a &lt;em>game-specific mapping override&lt;/em>.
This allows you to keep your usual mapping, but change it if you are more
comfortable with another mapping for another game; for example, a platformer
which is best played with Y and B, or a re-release of a game from the SFC/SNES
which you would like to play with the original controller arrangement.&lt;/p>

&lt;p>Buttons are sent to the GBA after mappings are applied.&lt;/p>

&lt;p>The default is to map DS buttons to the like-named GBA buttons:
A to A, B to B, L to L, R to R, Start to Start and Select to Select. The DS
buttons X and Y are unused; you can set them to be rapid fire buttons or use
them as hotkeys.&lt;/p>

&lt;p>When a game-specific mapping is present, it cancels a global mapping for the
same GBA button, &lt;em>not&lt;/em> the same DS button.&lt;/p>

&lt;p>Here's an example:&lt;/p>

&lt;p>&lt;code>Global mappings                     |  Game-specific overrides&lt;/code>&lt;/p>

&lt;p>&lt;code>GBA key A      = DS key A           |  GBA key A      = DS key B&lt;/code>&lt;/p>

&lt;p>&lt;code>GBA key B      = DS key B           |  GBA key B      = DS key Y&lt;/code>&lt;/p>

&lt;p>&lt;code>GBA key Start  = DS key Start       |  GBA key Start  = No override&lt;/code>&lt;/p>

&lt;p>&lt;code>GBA key Select = DS key Select      |  GBA key Select = No override&lt;/code>&lt;/p>

&lt;p>&lt;code>GBA key L      = DS key L           |  GBA key L      = DS key R&lt;/code>&lt;/p>

&lt;p>&lt;code>GBA key R      = DS key R           |  GBA key R      = No override&lt;/code>&lt;/p>

&lt;p>&lt;code>Rapid fire A   = DS key X           |  Rapid fire A   = No override&lt;/code>&lt;/p>

&lt;p>&lt;code>Rapid fire B   = DS key Y           |  Rapid fire B   = No override&lt;/code>&lt;/p>

&lt;p>Here's what each DS button would do in this example:&lt;/p>

&lt;ul>&lt;li>DS button A would not send any GBA buttons.&lt;/li>
&lt;li>DS button B would send GBA button A.&lt;/li>
&lt;li>DS button X would send GBA button A in rapid fire mode.&lt;/li>
&lt;li>DS button Y would send GBA button B &lt;strong>and&lt;/strong> GBA button B in rapid fire mode.
The behavior is undefined in this case; it may decide to send button B always
or only in rapid fire mode. The solution is to override &lt;em>Rapid fire B&lt;/em>.&lt;/li>
&lt;li>DS button L would not send any GBA buttons.&lt;/li>
&lt;li>DS button R would send GBA buttons L &lt;strong>and&lt;/strong> R. You may want to do this, but
if you didn't, the solution would be to map DS button L to GBA button R or
remove the override on GBA button L.&lt;/li>
&lt;li>Select and Start behave as expected.&lt;/li>

&lt;p>Transdroid Torrent Search is an Android application that provides torrent search results to other Android apps by exposing &lt;code>ContentProvider&lt;/code>s. Originally part of &lt;a href="http://www.transdroid.org">Transdroid&lt;/a>, it now supplies torrent links for 10+ torrent sites to various torrent-related Android applications.&lt;/p>

&lt;p>The latest .apk is available via &lt;a href="http://transdroid.org/latest-search">transdroid.org/latest-search&lt;/a> and code is available under the Lesser GPL v3 license.&lt;/p>

&lt;h1>&lt;a id="user-content-usage" class="anchor" href="#usage" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Usage&lt;/h1>

&lt;p>Transdroid Torrent Search provides access to torrent searches on a variety of sites. Instead of providing an interface itself, it allows Android application to access the data via a content provider.&lt;/p>

&lt;h2>&lt;a id="user-content-getting-search-results" class="anchor" href="#getting-search-results" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Getting search results&lt;/h2>

&lt;p>Acquiring search results for a specific query can be as easy as as two lines of code:&lt;/p>

&lt;pre>&lt;code>Uri uri = Uri.parse("content://org.transdroid.search.torrentsearchprovider/search/" + query);
Cursor results = managedQuery(uri, null, null, null, null);
&lt;/code>&lt;/pre>

&lt;p>The returned &lt;code>Cursor&lt;/code> can be used in a &lt;code>ListActivity&lt;/code> or elsewhere. The following fields are available in the returned cursor:&lt;/p>

&lt;pre>&lt;code>String[] fields = new String[] { "_ID", "NAME", "TORRENTURL", "DETAILSURL", "SIZE", "ADDED", "SEEDERS", "LEECHERS" };
&lt;/code>&lt;/pre>

&lt;p>&lt;em>Important:&lt;/em> Querying the content providers is an synchronous operation. If done from your application's UI thread this operation will stall the interface. Use, for example, an &lt;code>AsyncTask&lt;/code> to implement proper threading.&lt;/p>

&lt;h2>&lt;a id="user-content-customizing-search-results" class="anchor" href="#customizing-search-results" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Customizing search results&lt;/h2>

&lt;p>You have control over the search results that are returned. A specific site may be queried and the preferred sort order can be given:&lt;/p>

&lt;pre>&lt;code>Uri uri = Uri.parse("content://org.transdroid.search.torrentsearchprovider/search/" + query);
Cursor results = managedQuery(uri, null, "SITE = ?", new String[] { siteCode }, sortOrder)
&lt;/code>&lt;/pre>

&lt;p>Here, &lt;code>siteCode&lt;/code> is the code of one of the supported torrent sites. The default is &lt;code>Mininova&lt;/code>. The &lt;code>orderCode&lt;/code> is either BySeeders (default) or Combined. Note that no errors are returned when a site or sort order doesn't exist (although they are written to LogCat); an null &lt;code>Cursor&lt;/code> is returned instead. (This is a limitation of &lt;code>ContentResolver&lt;/code>s.)&lt;/p>

&lt;h2>&lt;a id="user-content-supported-torrent-sites" class="anchor" href="#supported-torrent-sites" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Supported torrent sites&lt;/h2>

&lt;p>To get a listing of (the codes of) the support torrent sites, you may use another provider:&lt;/p>

&lt;pre>&lt;code>uri = Uri.parse("content://org.transdroid.search.torrentsitesprovider/sites");
Cursor sites = managedQuery(uri, null, null, null, null);
&lt;/code>&lt;/pre>

&lt;p>The returned &lt;code>Cursor&lt;/code> contains the following fields:&lt;/p>

&lt;pre>&lt;code>String[] fields = new String[] { "_ID", "CODE", "NAME", "RSSURL" };
&lt;/code>&lt;/pre>

&lt;h2>&lt;a id="user-content-motivation-and-development-process" class="anchor" href="#motivation-and-development-process" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Motivation and development process&lt;/h2>

&lt;p>I'm a huge fan of Snapchat, a photo/video sharing app that allows you to set
expiration times on the media you send to your friends. They can't open it
after they've seen it for up to 10 seconds, and if they take a screenshot, the
other party is notified.&lt;/p>

&lt;p>I'm stunned and delighted by the fact that a simple
feature like auto-expiration of images can create such a compelling and
challenging service. And it's not just me: everyone I've told about Snapchat
who has used it has loved it, and as of last November more than one billion
snaps had been exchanged using the service.&lt;/p>

&lt;p>But I hate closed products, so I set about figuring out how it worked. &lt;a href="http://adamcaudill.com/2012/06/16/snapchat-api-and-security/">Adam
Caudill&lt;/a> wrote an
excellent analysis of their HTTP-based API by using an HTTPS traffic sniffer.
Unfortunately this information now seems out of date. &lt;/p>

&lt;p>I ended up having to fetch the official Android client's app binary (APK),
decompiling the whole thing with a mix of tools (all of them seemed to produce
subtly incorrect output), tracing the control flow a bit, and then puzzling
through the process of creating their dreaded access tokens (called req_token
in the HTTP calls).&lt;/p>

&lt;p>This involved me paging through Fiddler, trying to generate SHA-256 hashes
seemingly at random, tearing my heart out, and weeping openly.&lt;/p>

&lt;p>Their system is a bit unusual: it AES-256 hashes two input values separately,
using a secret key contained in the binary, and then uses a fixed pattern
string to pull bytes from one or the other. The final composition of the two is
used in HTTP requests. Why not just append the values pre-hash? The security
profile would be similar.&lt;/p>

&lt;p>Other things about the API that I've discovered so far:&lt;/p>

&lt;ul>&lt;li&gt;Speaks JSON over HTTPS, using POST as the verb&lt;/li>
&lt;li>Not made for human consumption; difficult error messaging&lt;/li>
&lt;li>Doesn't seem to support JSONP (i.e., callback parameter in post data is
ignored)&lt;/li>
&lt;li>Blob (image/video) downloads are encrypted using AES. This code successfully
decodes them before they are returned by the library. &lt;/li>
&lt;/ul>&lt;h2>&lt;a id="user-content-the-apocalyptic-future" class="anchor" href="#the-apocalyptic-future" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>The apocalyptic future&lt;/h2>

Made (Somewhat) Sensible.&lt;/h1>

&lt;p>Taking pictures or videos using a third-party app is fairly straightforward,
using &lt;code>ACTION_IMAGE_CAPTURE&lt;/code> or &lt;code>ACTION_VIDEO_CAPTURE&lt;/code>. However, you as the
developer have little control over what happens with the image or video,
other than indicating where the result gets stored. Plus, different camera
apps have slightly different behavior, meaning that you are prone to getting
inconsistent results.&lt;/p>

&lt;p>Taking pictures or videos using the built-in &lt;code>Camera&lt;/code> class directly is
eminently possible, but is full of edge and corner cases, not to mention
its own set of per-device idiosyncracies. As a result, a ton of code is
required to successfully show a preview, take a picture, and take a video.&lt;/p>

&lt;p>&lt;code>CWAC-Camera&lt;/code> is an effort to standardize that "ton of code" and hide it
behind a scalable API. Here, "scalable" means "simple things are simple,
but complex things may be a bit complex".&lt;/p>

&lt;p>In addition to what is written here,
JavaDocs &lt;a href="http://javadocs.commonsware.com/cwac/camera/index.html">are also available&lt;/a>.&lt;/p>

&lt;h2>&lt;a id="user-content-library-objectives" class="anchor" href="#library-objectives" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Library Objectives&lt;/h2>

&lt;p>The #1 objective of this library is maximum compatibility with hardware. As such,
this library will not be suitable for all use cases.&lt;/p>

&lt;p>The targeted use case is an app that might otherwise have relied upon
&lt;code>ACTION_IMAGE_CAPTURE&lt;/code>, but needs greater reliablilty and somewhat greater
control (e.g., capture images directly to internal storage).&lt;/p>

&lt;p>If you are trying to write "a camera app" — an app whose primary job is
to take pictures — this library may be unsuitable for you.&lt;/p>

&lt;h2>&lt;a id="user-content-basic-usage" class="anchor" href="#basic-usage" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Basic Usage&lt;/h2>

&lt;p>Step #1: Install the JARs or AARs as described above.&lt;/p>

&lt;p>Step #2: Add a &lt;code>CameraFragment&lt;/code> to your UI. You have two versions of
&lt;code>CameraFragment&lt;/code> to choose from:&lt;/p>

&lt;ul>&lt;li>&lt;p>&lt;code>com.commonsware.cwac.camera.CameraFragment&lt;/code> for use with native
API Level 11+ fragments&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>com.commonsware.cwac.camera.acl.CameraFragment&lt;/code> for use with
the Android Support package's backport of fragments and
&lt;a href="http://actionbarsherlock.com/">ActionBarSherlock&lt;/a>, supporting API Level 9
and 10&lt;/p>&lt;/li>
&lt;/ul>&lt;p>(note: if you choose the latter, your project will also need
to have the ActionBarSherlock library project)&lt;/p>

&lt;p>The &lt;code>CameraFragment&lt;/code> is responsible for rendering your preview, so
you need to size and position it as desired.&lt;/p>

&lt;p>Step #3: Call &lt;code>takePicture()&lt;/code> on the &lt;code>CameraFragment&lt;/code> when you want
to take a picture, which will be stored in the default digital photos
directory (e.g., &lt;code>DCIM&lt;/code>) on external storage as &lt;code>Photo_yyyyMMdd_HHmmss.jpg&lt;/code>, where
&lt;code>yyyyMMdd_HHmmss&lt;/code> is replaced by the current date and time. Note
that &lt;code>takePicture()&lt;/code> can throw an &lt;code>IllegalStateException&lt;/code> if you
call it before the preview is ready or if you call it while auto-focus
is occurring.&lt;/p>

&lt;p>Step #3b: Call &lt;code>startRecording()&lt;/code> and &lt;code>stopRecording()&lt;/code> on the
&lt;code>CameraFragment&lt;/code> to record a video. &lt;strong>NOTE&lt;/strong> that this is presently
only available on &lt;code>com.commonsware.cwac.camera.CameraFragment&lt;/code>
for use with native API Level 11+ fragments. The resulting video
will be stored in the default videos directory (e.g., &lt;code>Movies&lt;/code>) on external storage as
 &lt;code>Video_yyyyMMdd_HHmmss.mp4&lt;/code>, where
&lt;code&gt;yyyyMMdd_HHmmss&lt;/code> is replaced by the current date and time.&lt;/p>

&lt;p>Step #4: Add appropriate &lt;code>&amp;lt;uses-permission&amp;gt;&lt;/code> elements to your manifest.
For what is described in the preceding steps, you would need the &lt;code>CAMERA&lt;/code>,
&lt;code>RECORD_AUDIO&lt;/code>, and &lt;code>WRITE_EXTERNAL_STORAGE&lt;/code> permissions. &lt;code>RECORD_AUDIO&lt;/code>
is for the video recording using &lt;code>startRecording()&lt;/code>; if you are only taking
still photos, you will not need that permission.&lt;/p>

&lt;p>And that's it.&lt;/p>

&lt;p>&lt;code>CameraFragment&lt;/code> (and its underlying &lt;code>CameraView&lt;/code>)
will handle:&lt;/p>

&lt;ul>&lt;li>&lt;p>Showing the preview using an optimal preview frame size, and
managing the aspect ratio of the on-screen preview &lt;code>View&lt;/code> so
that your previews do not appear stretched&lt;/p>&lt;/li>
&lt;li>&lt;p>Dealing with configuration changes and screen rotation, so
your camera activity can work in portrait or landscape&lt;/p>&lt;/li>
&lt;li>&lt;p>Following the appropriate recipes for taking still pictures
and videos, including choosing the largest-available image size
for the resolution&lt;/p>&lt;/li>
&lt;li>&lt;p>Opening and closing the camera at the appropriate times, so
when you are in the foreground you have exclusive camera access,
but other apps will have access to the camera while your activity
is not in the foreground&lt;/p>&lt;/li>
&lt;li>&lt;p>And more!&lt;/p>&lt;/li>
&lt;/ul>&lt;h2>&lt;a id="user-content-simple-configuration-and-usage" class="anchor" href="#simple-configuration-and-usage" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Simple Configuration and Usage&lt;/h2>

&lt;p>Of course, there are probably plenty of things that you will want to configure
about the process of taking photos and videos. There are many hooks in &lt;code>CWAC-Camera&lt;/code>
to allow you to do just that.&lt;/p>

&lt;p>Much of this configuration involves creating a custom &lt;code>CameraHost&lt;/code>. &lt;code>CameraHost&lt;/code>
is your primary interface with the &lt;code>CWAC-Camera&lt;/code> classes for configuring
the behavior of the camera. &lt;code>CameraHost&lt;/code> is an interface, one that you are
welcome to implement in full. Most times, though, you will be better served
extending &lt;code>SimpleCameraHost&lt;/code>, the default implementation of &lt;code>CameraHost&lt;/code>,
so that you can override only those methods where you want behavior different
from the default.&lt;/p>

&lt;p>&lt;code>SimpleCameraHost&lt;/code> also offers &lt;code>SimpleCameraHost.Builder&lt;/code>
that you can use for some simple configuration, instead of creating a subclass
of &lt;code>SimpleCameraHost&lt;/code> and overriding methods. Create an instance of the &lt;code>Builder&lt;/code>
(passing in a &lt;code>Context&lt;/code> to the constructor, such as your activity), call the
various builder-style setters for configuration, and call &lt;code>build()&lt;/code> to get your
customized &lt;code>SimpleCameraHost&lt;/code> instance.&lt;/p>

&lt;p>You can pass your customized instance of &lt;code>CameraHost&lt;/code>
to &lt;code>setHost()&lt;/code> on your &lt;code>CameraFragment&lt;/code>, to replace the default.
&lt;strong>Do this in &lt;code>onCreate()&lt;/code> of a &lt;code>CameraFragment&lt;/code> subclass&lt;/strong> (or, if practical,
just after instantiating your fragment) to ensure that the
right &lt;code>CameraHost&lt;/code> is used everywhere.&lt;/p>

&lt;h3>&lt;a id="user-content-picturetransaction" class="anchor" href="#picturetransaction" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>PictureTransaction&lt;/h3>

&lt;p>The &lt;code>takePicture()&lt;/code> method has a zero-argument version that just takes a picture.
It also has a one-argument version, where the argument is an instance of
&lt;code>PictureTransaction&lt;/code>. This will allow you to configure details of this particular
picture to be taken. You can create a &lt;code>PictureTransaction&lt;/code> using its constructor,
which takes an instance of your &lt;code>CameraHost&lt;/code> as a parameter.&lt;/p>

&lt;p>Various sections below will mention using builder-style setters on &lt;code>PictureTransaction&lt;/code>
to control how a picture is taken.&lt;/p>

&lt;p>&lt;code>PictureTransaction&lt;/code> has a &lt;code>tag(Object)&lt;/code> method that allow you to attach
arbitrary data to the transaction, without having to bother with a subclass. You can
retrieve that object later via the zero-argument &lt;code>tag()&lt;/code> method.&lt;/p>

&lt;h3>&lt;a id="user-content-controlling-the-names-and-locations-of-output-files" class="anchor" href="#controlling-the-names-and-locations-of-output-files" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Controlling the Names and Locations of Output Files&lt;/h3>

&lt;p>There are a series of methods that you can override on &lt;code>SimpleCameraHost&lt;/code>
to control where photos and videos
are stored once taken. These methods will be called for each &lt;code>takePicture()&lt;/code>
or &lt;code>startRecording()&lt;/code> call, so you can create customized results for each
distinct photo or video.&lt;/p>

&lt;p>Specifically:&lt;/p>

&lt;ul>&lt;li>&lt;p>Override &lt;code>getPhotoFilename()&lt;/code> to return the base name of the file to use
to store the photo&lt;/p>&lt;/li>
&lt;li>&lt;p>Override &lt;code>getPhotoDirectory()&lt;/code> to return the name of the directory in which
to store the photo&lt;/p>&lt;/li>
&lt;li>&lt;p>Override &lt;code>getPhotoPath()&lt;/code> to return the complete &lt;code>File&lt;/code> object pointing
to the desired file in the desired directory (the default implementation
combines the results of &lt;code>getPhotoDirectory()&lt;/code> and &lt;code>getPhotoFilename()&lt;/code>, so
overriding &lt;code>getPhotoPath()&lt;/code> replaces all of that)&lt;/p>&lt;/li>
&lt;/ul>&lt;p>There are equivalent &lt;code>getVideoFilename()&lt;/code>, &lt;code>getVideoDirectory()&lt;/code>, and
&lt;code>getVideoPath()&lt;/code> for controlling the output of the next video to be taken.&lt;/p>

&lt;p>&lt;code>SimpleCameraHost.Builder&lt;/code> offers &lt;code>photoDirectory()&lt;/code> and &lt;code>videoDirectory()&lt;/code>
setters, where you provide the &lt;code>File&lt;/code> pointing to your desired directory.&lt;/p>

&lt;p>By default, if you are using &lt;code>SimpleCameraHost&lt;/code>, your image will be indexed
by the &lt;code>MediaStore&lt;/code>. If you do not want this, override &lt;code>scanSavedImage()&lt;/code>
to return &lt;code>false&lt;/code> in your &lt;code>SimpleCameraHost&lt;/code> subclass (or, call &lt;code>scanSavedImage()&lt;/code>
and pass in a &lt;code>boolean&lt;/code> to use by default). This is called on a
per-image basis.&lt;/p>

&lt;h3>&lt;a id="user-content-controlling-which-camera-is-used" class="anchor" href="#controlling-which-camera-is-used" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Controlling Which Camera is Used&lt;/h3>

&lt;p>If you override &lt;code>useFrontFacingCamera()&lt;/code> on &lt;code>SimpleCameraHost&lt;/code> to return
&lt;code>true&lt;/code>, the front-facing camera will be used, instead of the default rear-facing
camera. You can also call &lt;code>useFrontFacingCamera()&lt;/code> on your
&lt;code>SimpleCameraHost.Builder&lt;/code>, passing in a &lt;code>boolean&lt;/code>
default value to use.&lt;/p>

&lt;p>Or, override &lt;code>getDeviceId()&lt;/code> (available on &lt;code>CameraHost&lt;/code>), and you can provide
the ID of the specific camera you want. This would involve your choosing an
available camera based on your own criteria. See the JavaDocs for Android's
&lt;code>Camera&lt;/code> class, notably
&lt;a href="http://developer.android.com/reference/android/hardware/Camera.html#getNumberOfCameras()">&lt;code>getNumberOfCameras()&lt;/code>&lt;/a>
and &lt;a href="http://developer.android.com/reference/android/hardware/Camera.html#getCameraInfo(int,%20android.hardware.Camera.CameraInfo)">&lt;code>getCameraInfo()&lt;/code>&lt;/a>
for more. You can also call &lt;code>deviceId()&lt;/code> on &lt;code>SimpleCameraHost&lt;/code> to supply
the camera ID to use.&lt;/p>

&lt;h3>&lt;a id="user-content-controlling-ffc-mirror-correction" class="anchor" href="#controlling-ffc-mirror-correction" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Controlling FFC Mirror Correction&lt;/h3>

&lt;p>By default, the pictures taken from the front-facing camera are a mirror
image of what is shown on the preview. If you wish for the front-facing
camera photos to match the preview, override &lt;code>mirrorFFC()&lt;/code> on your &lt;code>CameraHost&lt;/code>
and have it
return &lt;code>true&lt;/code>, and &lt;code>CWAC-Camera&lt;/code> will reverse the image for you before
saving it. Or, call &lt;code>mirrorFFC()&lt;/code> on your &lt;code>SimpleCameraHost.Builder&lt;/code>, supplying a &lt;code>boolean&lt;/code>
value to use as the default. Or, call &lt;code>mirrorFFC()&lt;/code> on your &lt;code>PictureTransaction&lt;/code>,
to control this for an individual picture.&lt;/p>

&lt;h3>&lt;a id="user-content-handling-exceptions" class="anchor" href="#handling-exceptions" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Handling Exceptions&lt;/h3>

&lt;p>There are some exceptions that are thrown by the &lt;code>Camera&lt;/code> class (and kin, like
&lt;code>MediaRecorder&lt;/code>). Those are passed to your host's &lt;code>handleException()&lt;/code>
method. The default implementation displays a &lt;code>Toast&lt;/code> and logs the message
to LogCat as an error, but you probably will want to replace that with
something else that integrates better with your UI.&lt;/p>

&lt;h3>&lt;a id="user-content-supporting-full-bleed-preview" class="anchor" href="#supporting-full-bleed-preview" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Supporting "Full-Bleed Preview"&lt;/h3>

&lt;p>The original default behavior of &lt;code>CameraFragment&lt;/code> and &lt;code>CameraView&lt;/code> was to show the entire
preview, as supplied by the underlying &lt;code>Camera&lt;/code> API. Since the aspect ratio of
the preview frames may be different than the aspect ratio of the &lt;code>CameraView&lt;/code>,
this results in a "letterbox" effect, where the background will show through on
one axis on the sides.&lt;/p>

&lt;p>The new default behavior is to completely fill the &lt;code>CameraView&lt;/code>, at the cost of
cropping off some of the actual preview frame, what is known as "full-bleed preview"
(stealing some terminology from the world of print media).&lt;/p>

&lt;p>To control this behavior:&lt;/p>

&lt;ul>&lt;li>&lt;p>Have your &lt;code>CameraHost&lt;/code>
return &lt;code>true&lt;/code> or &lt;code>false&lt;/code> from &lt;code>useFullBleedPreview()&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Or, call &lt;code>useFullBleedPreview()&lt;/code>
on your &lt;code>SimpleCameraHost.Builder&lt;/code>, passing in a &lt;code>boolean&lt;/code> value to use by default.&lt;/p>&lt;/li>
&lt;/ul>&lt;p>Note that the pictures and videos taken by this library are unaffected by
&lt;code>useFullBleedPreview()&lt;/code>. Hence, if &lt;code>useFullBleedPreview()&lt;/code> returns &lt;code>true&lt;/code>, the
picture or video may contain additional content on the edges that was not
visible in the preview.&lt;/p>

&lt;h3>&lt;a id="user-content-wrapping-the-preview-ui" class="anchor" href="#wrapping-the-preview-ui" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Wrapping the Preview UI&lt;/h3>

&lt;p>From a UI standpoint, the &lt;code>CameraFragment&lt;/code> solely handles the preview pane.
Presumably, you will need
more to your UI than this, such as buttons to allow users to take pictures or
record videos. You have two major options here:&lt;/p>

&lt;ol>&lt;li>&lt;p>You can put that UI as a peer to the &lt;code>CameraFragment&lt;/code>, such as by having action
bar items, as the demo apps do.&lt;/p>&lt;/li>
&lt;li>&lt;p>You can subclass &lt;code>CameraFragment&lt;/code> and override &lt;code>onCreateView()&lt;/code>. Chain to the
superclass to get the &lt;code>CameraFragment&lt;/code>'s own UI, then wrap that in your own
container with additional widgets, and return the combined UI from your &lt;code>onCreateView()&lt;/code>.
You can see this in the main demo app, which adds a &lt;code>SeekBar&lt;/code> or &lt;code>VerticalSeekBar&lt;/code>
for zoom levels.&lt;/p>&lt;/li>
&lt;/ol>&lt;p>It is also possible to replace &lt;code>onCreateView()&lt;/code> completely with your own
implementation, or otherwise use &lt;code>CameraView&lt;/code> from a layout resource. This is
covered later in this document.&lt;/p>

&lt;h3>&lt;a id="user-content-auto-focus" class="anchor" href="#auto-focus" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Auto-Focus&lt;/h3>

&lt;p>You can call &lt;code>autoFocus()&lt;/code> on &lt;code>CameraFragment&lt;/code> or &lt;code>CameraView&lt;/code> to trigger any
auto-focus behavior that you have configured via &lt;code>setFocusMode()&lt;/code> on &lt;code>Camera.Parameters&lt;/code>.
You can call
&lt;code>cancelAutoFocus()&lt;/code> on &lt;code>CameraFragment&lt;/code> or &lt;code>CameraView&lt;/code> to ensure that auto-focus
mode has been canceled.&lt;/p>

&lt;p>Note that auto-focus is only available in certain conditions, notably when
the preview mode is enabled. You can call &lt;code>isAutoFocusAvailable()&lt;/code> on &lt;code>CameraFragment&lt;/code>
or &lt;code>CameraView&lt;/code> to determine if auto-focus is presently available for use. Calling
&lt;code>autoFocus()&lt;/code> when auto-focus is not available will have no effect.&lt;/p>

&lt;p>&lt;code>CameraHost&lt;/code> implementations will need to implement an &lt;code>onAutoFocus()&lt;/code> method, coming
from
&lt;a href="https://developer.android.com/reference/android/hardware/Camera.AutoFocusCallback.html">the &lt;code>Camera.AutoFocusCallback&lt;/code> interface&lt;/a>
that &lt;code>CameraHost&lt;/code> extends.
&lt;code>SimpleCameraHost&lt;/code> has a default implementation of &lt;code>onAutoFocus()&lt;/code> that
plays a
device-standard sound upon completion (API Level 16+ only).&lt;/p>

&lt;p>&lt;code>CameraHost&lt;/code> implementations will also need &lt;code>autoFocusAvailable()&lt;/code> and
&lt;code>autoFocusUnavailable()&lt;/code> methods, to be notified when auto-focus is available or
not. This can be used to trigger whether action bar items are enabled, etc.
&lt;code>SimpleCameraHost&lt;/code> has no-op implementations of these callbacks.&lt;/p>

&lt;h3>&lt;a id="user-content-single-shot-mode" class="anchor" href="#single-shot-mode" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Single-Shot Mode&lt;/h3>

&lt;p>By default, the result of taking a picture is to return the &lt;code>CameraFragment&lt;/code>
to preview mode, ready to take the next picture. If, instead, you only need
the one picture, or you want to send the user to some other bit of UI first
and do not want preview to start up again right away, override
&lt;code>useSingleShotMode()&lt;/code> in your &lt;code>CameraHost&lt;/code> to return &lt;code>true&lt;/code>. Or, call
&lt;code>useSingleShotMode()&lt;/code> on your &lt;code>SimpleCameraHost.Builder&lt;/code>, passing in a &lt;code>boolean&lt;/code> to
use by default. Or, call &lt;code>useSingleShotMode()&lt;/code> on your &lt;code>PictureTransaction&lt;/code>,
to control this for an individual picture.&lt;/p>

&lt;p>You will then
probably want to use your own &lt;code>saveImage()&lt;/code> implementation in your
&lt;code>CameraHost&lt;/code> to do whatever you want instead of restarting the preview.
For example, you could start another activity to do something with the
image. However, bear in mind that an &lt;code>Intent&lt;/code> is limited to ~1MB, and so
passing an image to another activity via a &lt;code>Intent&lt;/code> extra is likely to be
unreliable. You will need to do something else, such as (carefully) use a
static data member.&lt;/p>

&lt;p>Preview mode will re-enable automatically after an &lt;code>onPause()&lt;/code>/&lt;code>onResume()&lt;/code>
cycle of your &lt;code>CameraFragment&lt;/code>, or you can call &lt;code>restartPreview()&lt;/code> on your
&lt;code>CameraFragment&lt;/code> (or &lt;code>CameraView&lt;/code>).&lt;/p>

&lt;h3>&lt;a id="user-content-zoom-support" class="anchor" href="#zoom-support" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Zoom Support&lt;/h3>

&lt;p>To zoom the camera, call &lt;code>zoomTo()&lt;/code> on the &lt;code>CameraView&lt;/code> or &lt;code>CameraFragment&lt;/code>,
supplying the integer zoom level that you want. This level must be between
0 and what &lt;code>Camera.Parameters&lt;/code> returns from &lt;code>getMaxZoom()&lt;/code>. The
&lt;code>adjustPreviewParameters()&lt;/code> callback method in your &lt;code>CameraHost&lt;/code> is a good
time to get this value and configure your UI (e.g., &lt;code>SeekBar&lt;/code>) to allow
the user to zoom the camera.&lt;/p>

&lt;p>&lt;code>zoomTo()&lt;/code> returns a &lt;code>ZoomTransaction&lt;/code>. This has a series of builder-style
methods (a.k.a., a fluent interface) that allow you to configure the
transaction, where the methods return the transaction so you can chain on 
the next call. The configuration methods are:&lt;/p>

&lt;ul>&lt;li>&lt;p>&lt;code>onComplete()&lt;/code> to supply a &lt;code>Runnable&lt;/code> to be executed when we have reached
the zoom level&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>onChange()&lt;/code> to supply a &lt;code>Camera.OnZoomChangeListener&lt;/code> to be called as
we progress to the desired zoom level&lt;/p>&lt;/li>
&lt;/ul>&lt;p>Once configured, call &lt;code>go()&lt;/code> to run the transaction.&lt;/p>

&lt;p>If the camera supports smooth zoom, the zoom transaction will take a few
moments, and you can cancel the operation by calling &lt;code>cancel()&lt;/code> on the
&lt;code>ZoomTransaction&lt;/code>. If the camera does not support smooth zoom, the zoom
level is just immediately changed.&lt;/p>

&lt;p>Note that your &lt;code>OnZoomChangeListener&lt;/code> supplied to &lt;code>onChange()&lt;/code> will be
called before the &lt;code>onComplete()&lt;/code> &lt;code>Runnable&lt;/code>, if you happen to supply both.&lt;/p>

&lt;p>The main demo app adds a &lt;code>SeekBar&lt;/code> and &lt;code>VerticalSeekBar&lt;/code> to control zoom
levels, so you can see how this is used.&lt;/p>

&lt;p>Note that some devices lie about their zoom capabilities. For example,
the Motorola RAZR i's front-facing camera apparently does not support
zoom, where &lt;code>getMaxZoom()&lt;/code> still returns a positive value.
&lt;code>doesZoomReallyWork()&lt;/code> on your &lt;code>CameraFragment&lt;/code> or &lt;code>CameraView&lt;/code> will
return &lt;code>false&lt;/code> if zoom is known to be broken for the current camera
on the current device. In this case, do not zoom, or your code may
go "boom".&lt;/p>

&lt;h3>&lt;a id="user-content-camera-fail" class="anchor" href="#camera-fail" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Camera? #FAIL&lt;/h3>

&lt;p>If &lt;code>getCameraId()&lt;/code> of your &lt;code>CameraHost&lt;/code> returns a negative value, &lt;code>CameraView&lt;/code>
will assume that there are no valid cameras (e.g., your app is running on a
game console). In addition to avoiding anything that tries to touch the camera,
your &lt;code>CameraHost&lt;/code> will be called with &lt;code>onCameraFail()&lt;/code>, where you will be
supplied with a &lt;code>FailureReason&lt;/code> of &lt;code>NO_CAMERAS_REPORTED&lt;/code>.&lt;/p>

&lt;p>If anything else goes wrong when trying to open the camera (e.g., a device admin
policy has disabled the camera), your &lt;code>onCameraFail()&lt;/code> method will be called
with a &lt;code>FailureReason&lt;/code> of &lt;code>UNKNOWN&lt;/code>.&lt;/p>

&lt;p>While &lt;code>SimpleCameraHost&lt;/code> has a trivial &lt;code>onCameraFail()&lt;/code> implementation (just
logging to LogCat), you are strongly encouraged to override this and inform
your users of the problem.&lt;/p>

&lt;h3>&lt;a id="user-content-fixing-up-images-and-your-heap" class="anchor" href="#fixing-up-images-and-your-heap" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Fixing Up Images... And Your Heap&lt;/h3>

&lt;p>There are a few fixups that the library performs on your images, 
to provide consistent output. The biggest one is to rotate the image
to the proper orientation, rather than rely on EXIF headers, as not
all image viewers use those headers.&lt;/p>

&lt;p>The problem is that these fixups take a lot of heap space. By default,
the library will always try to perform these fixups, and for a
high-resolution image on a low-memory device, an &lt;code>OutOfMemoryError&lt;/code>
may result.&lt;/p>

&lt;p>You have two means of managing this, and you are welcome to apply
one or both of them:&lt;/p>

&lt;ol>&lt;li>&lt;p>You can add &lt;code>android:largeHeap="true"&lt;/code> to the &lt;code>&amp;lt;application&amp;gt;&lt;/code>
element of your manifest. On API Level 11+ devices, you will get
more heap space, making it more likely that the fixup will
succeed. However, this will hurt the user, in that your app will
tend to kick other apps out of memory more quickly, which the user
may not appreciate.&lt;/p>&lt;/li>
&lt;li>&lt;p>Your &lt;code>CameraHost&lt;/code> can return a value between &lt;code>0.0f&lt;/code> and &lt;code>1.0f&lt;/code>
from &lt;code>maxPictureCleanupHeapUsage()&lt;/code>. The default implementation
on &lt;code>SimpleCameraHost&lt;/code> returns &lt;code>1.0f&lt;/code>, which says "the &lt;code>byte[]&lt;/code>
of image data from the camera can be as big as our heap limit,
and we should still try to do the cleanup work". A value of &lt;code>0.0f&lt;/code>
would indicate that the cleanup work should never be done, and
the images will be saved in their natural state. A value in
between represents a portion of the heap space; if the &lt;code>byte[]&lt;/code>
is that size or smaller, go ahead and try to do the fixups. Note
that this determination is made on the compressed JPEG &lt;code>byte[]&lt;/code>
length, not the size of the decoded &lt;code>Bitmap&lt;/code>, and the JPEG may
be compressed ~90% compared to its uncompressed size.&lt;/p>&lt;/li>
&lt;/ol>&lt;h2>&lt;a id="user-content-advanced-configuration" class="anchor" href="#advanced-configuration" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Advanced Configuration&lt;/h2>

&lt;p>In addition to the configuration hooks specified above, you can do more
to tailor how photos and videos are taken.&lt;/p>

&lt;h3>&lt;a id="user-content-controlling-preview-sizes" class="anchor" href="#controlling-preview-sizes" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Controlling Preview Sizes&lt;/h3>

&lt;p>Your &lt;code>CameraHost&lt;/code> will be called with &lt;code>getRecordingHint()&lt;/code>, to determine if the preview
should be optimized for possible video recording, or not (i.e., only still images will
be taken). You can return a &lt;code>CameraHost.RecordingHint&lt;/code> enum: &lt;code>STILL_ONLY&lt;/code>, &lt;code>VIDEO_ONLY&lt;/code>,
or &lt;code>ANY&lt;/code>.&lt;/p>

&lt;p>Usually, your &lt;code>CameraHost&lt;/code> will be called with &lt;code>getPreviewSize()&lt;/code>, where you need to return
a valid &lt;code>Camera.Size&lt;/code> indicating the desired size of the preview frames. &lt;code>getPreviewSize()&lt;/code>
is passed:&lt;/p>

&lt;ul>&lt;li>&lt;p>the display orientation, in degrees, with 0 indicating landscape, 90 indicating
portrait, etc.&lt;/p>&lt;/li>
&lt;li>&lt;p>the available width and height for the preview&lt;/p>&lt;/li>
&lt;li>&lt;p>the &lt;code>Camera.Parameters&lt;/code> object, from which you can determine the valid preview sizes
by calling &lt;code>getSupportedPreviewSizes()&lt;/code>&lt;/p>&lt;/li>
&lt;/ul>&lt;p>The &lt;code>CameraUtils&lt;/code> class contains three static methods with stock algorithms for
choosing the preview size:&lt;/p>

&lt;ol>&lt;li>&lt;p>&lt;code>getOptimalPreviewSize()&lt;/code> uses the algorithm found in the SDK camera sample app&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>getBestAspectPreviewSize()&lt;/code> finds the preview size that most closely matches the
aspect ratio of our available space&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>getBestAspectPreviewSize(double)&lt;/code> finds the preview size that offers the biggest
preview size that only differs from the desired aspect ratio by the supplied
&lt;code>closeEnough&lt;/code> value (&lt;code>closeEnough&lt;/code> of &lt;code>0.0d&lt;/code> would give the same results as does
&lt;code>getBestAspectPreviewSize()&lt;/code>)&lt;/p>&lt;/li>
&lt;/ol>&lt;p>&lt;code>SimpleCameraHost&lt;/code> uses &lt;code>getBestAspectPreviewSize()&lt;/code> for the default implementation
of &lt;code>getPreviewSize()&lt;/code>. You can override &lt;code>getPreviewSize()&lt;/code> and substitute in your
own selection algorithm. Just make sure that the returned size is one of the ones
returned by &lt;code>getSupportedPreviewSizes()&lt;/code>.&lt;/p>

&lt;p>If &lt;code>getRecordingHint()&lt;/code> returns &lt;code>ANY&lt;/code> or &lt;code>VIDEO_ONLY&lt;/code>, though,
&lt;code>CameraHost&lt;/code> supplies the preview
size via &lt;code>getPreferredPreviewSizeForVideo()&lt;/code> instead of &lt;code>getPreviewSize()&lt;/code>. If you
wish to use a different preview size for video, return it, otherwise return &lt;code>null&lt;/code>
and we will use the results from &lt;code>getPreviewSize()&lt;/code> instead. &lt;code>getPreferredPreviewSizeForVideo()&lt;/code>
is passed a &lt;code>Camera.Size&lt;/code> as a hint from the device for a value to use, instead of
anything you might get yourself from &lt;code>Camera.Parameters&lt;/code> -- while using the hinted
value is probably a good idea (if it is not &lt;code>null&lt;/code>), it is not required.&lt;/p>

&lt;h3>&lt;a id="user-content-controlling-picture-sizes" class="anchor" href="#controlling-picture-sizes" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Controlling Picture Sizes&lt;/h3>

&lt;p>Similarly, your &lt;code>CameraHost&lt;/code> will be called with &lt;code>getPictureSize()&lt;/code>, for you to return
the desired &lt;code>Camera.Size&lt;/code> of the still images taken by the camera. You are simply passed the
&lt;code>Camera.Parameters&lt;/code>, on which you can call &lt;code>getSupportedPictureSizes()&lt;/code> to find out
the possible picture sizes that you can choose from.&lt;/p>

&lt;p>The &lt;code>CameraUtils&lt;/code> class has a pair of methods for simple algorithms for choosing a picture
size:&lt;/p>

&lt;ol>&lt;li>&lt;p>&lt;code>getLargestPictureSize()&lt;/code> returns the &lt;code>Camera.Size&lt;/code> that is the largest in area&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>getSmallestPictureSize()&lt;/code> returns the &lt;code>Camera.Size&lt;/code> that is the smallest in area&lt;/p>&lt;/li>
&lt;/ol>&lt;p>&lt;code>SimpleCameraHost&lt;/code> uses &lt;code>getLargestPictureSize()&lt;/code> for the default implementation
of &lt;code>getPictureSize()&lt;/code>. You can override &lt;code>getPictureSize()&lt;/code> and substitute in your
own selection algorithm. Just make sure that the returned size is one of the ones
returned by &lt;code>getSupportedPictureSizes()&lt;/code>.&lt;/p>

&lt;h3>&lt;a id="user-content-arbitrary-preview-configuration" class="anchor" href="#arbitrary-preview-configuration" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Arbitrary Preview Configuration&lt;/h3>

&lt;p>When setting up the camera preview, your &lt;code>CameraHost&lt;/code> will be called with
&lt;code>adjustPreviewParameters()&lt;/code>, passing in a &lt;code>Camera.Parameters&lt;/code>. Here, you can make
any desired adjustments to the camera preview, &lt;em>except&lt;/em> the preview size (which you
should be handling in &lt;code>getPreviewSize()&lt;/code>). &lt;code>adjustPreviewParameters()&lt;/code> returns
the revised &lt;code>Camera.Parameters&lt;/code>, where the stock implementation in 
&lt;code>SimpleCameraHost&lt;/code> just returns the passed-in parameters unmodified.&lt;/p>

&lt;h3>&lt;a id="user-content-arbitrary-photo-configuration" class="anchor" href="#arbitrary-photo-configuration" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Arbitrary Photo Configuration&lt;/h3>

&lt;p>Shortly after you call &lt;code>takePicture()&lt;/code> on your &lt;code>CameraFragment&lt;/code>,
your &lt;code>CameraHost&lt;/code> will be called with
&lt;code>adjustPictureParameters()&lt;/code>, passing in a &lt;code>Camera.Parameters&lt;/code>. Here, you can make
any desired adjustments to the parameters related to taking photos,
&lt;em>except&lt;/em> the image size (which you
should be handling in &lt;code>getPictureSize()&lt;/code>). &lt;code>adjustPictureParameters()&lt;/code> returns
the revised &lt;code>Camera.Parameters&lt;/code>, where the stock implementation in 
&lt;code>SimpleCameraHost&lt;/code> just returns the passed-in parameters unmodified.&lt;/p>

&lt;h3>&lt;a id="user-content-arbitrary-video-configuration" class="anchor" href="#arbitrary-video-configuration" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Arbitrary Video Configuration&lt;/h3>

&lt;p>Shortly after you call &lt;code>startRecording()&lt;/code>, your &lt;code>CameraHost&lt;/code> will be called
with:&lt;/p>

&lt;ul>&lt;li>&lt;p>&lt;code>configureRecorderAudio()&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>configureRecorderProfile()&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>configureRecorderOutput()&lt;/code>&lt;/p>&lt;/li>
&lt;/ul>&lt;p>in that order. Here, you can help tailor the way videos get recorded.
Each of these is passed the ID of the camera being used for recording plus
the &lt;code>MediaRecorder&lt;/code> instance that does the actual recording.&lt;/p>

&lt;p>The stock &lt;code>SimpleCameraHost&lt;/code> does the following:&lt;/p>

&lt;ul>&lt;li>&lt;p>In &lt;code>configureRecorderAudio()&lt;/code>, &lt;code>SimpleCameraHost&lt;/code> calls
&lt;code>setAudioSource(MediaRecorder.AudioSource.CAMCORDER)&lt;/code> on the &lt;code>MediaRecorder&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>In &lt;code>configureRecorderProfile()&lt;/code>, &lt;code>SimpleCameraHost&lt;/code> calls
&lt;code>setProfile(CamcorderProfile.get(cameraId, CamcorderProfile.QUALITY_HIGH))&lt;/code>
on the &lt;code>MediaRecorder&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>In &lt;code>configureRecorderOutput()&lt;/code>, &lt;code>SimpleCameraHost&lt;/code> calls
&lt;code>setOutputFile(getVideoPath().getAbsolutePath())&lt;/code> on the &lt;code>MediaRecorder&lt;/code>
(where &lt;code>getVideoPath()&lt;/code> was described earlier in this document)&lt;/p>&lt;/li>
&lt;/ul>&lt;p>While these are reasonable defaults, you are welcome to override these
implementations to do something else.&lt;/p>

&lt;h3>&lt;a id="user-content-overriding-photo-saving" class="anchor" href="#overriding-photo-saving" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Overriding Photo Saving&lt;/h3>

&lt;p>The default &lt;code>SimpleCameraHost&lt;/code> logic for saving photos uses the &lt;code>getPhotoPath()&lt;/code> 
and related methods discussed above. Actually saving the photo is done in
&lt;code>saveImage(PictureTransaction, byte[])&lt;/code>, called on your &lt;code>CameraHost&lt;/code>, where &lt;code>SimpleCameraHost&lt;/code> has a
&lt;code>saveImage(PictureTransaction, byte[])&lt;/code> implementation that writes the supplied &lt;code>byte[]&lt;/code> out to the desired
location.&lt;/p>

&lt;p>You are welcome to override &lt;code>saveImage(PictureTransaction, byte[])&lt;/code> and do something else with the &lt;code>byte[]&lt;/code>, 
such as send it over the Internet. &lt;code>saveImage(PictureTransaction, byte[])&lt;/code> is called on a background thread,
so you do not have to do your own asynchronous work.&lt;/p>

&lt;p>Another use for this is to find out when the saving is complete, so that you can
use the resulting image. Just override &lt;code>saveImage(PictureTransaction, byte[])&lt;/code>, chain to the superclass
implementation, and when that returns, the image is ready for use.&lt;/p>

&lt;p>There is also a &lt;code>saveImage(PictureTransaction, Bitmap)&lt;/code> callback, giving you a decoded &lt;code>Bitmap&lt;/code>
instead of a &lt;code>byte[]&lt;/code>.&lt;/p>

&lt;p>By default, &lt;code>saveImage(PictureTransaction, byte[])&lt;/code> will be called, and not
&lt;code>saveImage(PictureTransaction, Bitmap)&lt;/code>. To change this, call &lt;code>needBitmap(boolean)&lt;/code>
and/or &lt;code>needByteArray(boolean)&lt;/code> on your &lt;code>PictureTransaction&lt;/code>, passing that
&lt;code>PictureTransaction&lt;/code> to &lt;code>takePicture()&lt;/code>.&lt;/p>

&lt;p>Note that if you say that you need the &lt;code>Bitmap&lt;/code>, you are
responsible for the &lt;code>Bitmap&lt;/code> (e.g., calling &lt;code>recycle()&lt;/code> on it) once it is handed
to your host.&lt;/p>

&lt;h3>&lt;a id="user-content-controlling-the-shutter-callback" class="anchor" href="#controlling-the-shutter-callback" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Controlling the Shutter Callback&lt;/h3>

&lt;p>You can subclass &lt;code>PictureTransaction&lt;/code> and override &lt;code>onShutter()&lt;/code> to
do something when the shutter is pressed.&lt;/p>

&lt;p>Also,
your &lt;code>CameraHost&lt;/code> implementation can return a &lt;code>Camera.ShutterCallback&lt;/code> object
via &lt;code>getShutterCallback()&lt;/code>,
which will be used in the underlying &lt;code>takePicture()&lt;/code> call on the Android &lt;code>Camera&lt;/code>,
giving you control to play a "shutter click" sound. &lt;code>SimpleCameraHost&lt;/code> returns &lt;code>null&lt;/code>
from &lt;code>getShutterCallback()&lt;/code>, to give you the device default behavior.&lt;/p>

&lt;h3>&lt;a id="user-content-detecting-faces" class="anchor" href="#detecting-faces" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Detecting Faces&lt;/h3>

&lt;p>If you wish to use the face detection APIs available on API Level 14+, do the
following:&lt;/p>

&lt;ol>&lt;li>&lt;p>Have your &lt;code>CameraHost&lt;/code> implementation also implement &lt;code>Camera.FaceDetectionListener&lt;/code>.&lt;/p>&lt;/li>
&lt;li>&lt;p>Override &lt;code>adjustPreviewParameters()&lt;/code> in your &lt;code>CameraHost&lt;/code> and take that opportunity
to check the value of &lt;code>getMaxNumDetectedFaces()&lt;/code>, a method on &lt;code>Camera.Parameters&lt;/code>.
If that returns 0, face detection is not supported by the device. &lt;strong>NOTE&lt;/strong>: a better
API for this may be added in the future.&lt;/p>&lt;/li>
&lt;li>&lt;p>Override &lt;code>autoFocusAvailable()&lt;/code> in your &lt;code>CameraHost&lt;/code>, and if face detection is
enabled, call &lt;code>startFaceDetection()&lt;/code> on your &lt;code>CameraFragment&lt;/code> or &lt;code>CameraView&lt;/code>.
&lt;strong>NOTE&lt;/strong>: a dedicated callback for this may be added in the future — this is
a stop-gap to allow this fix to go in a patch release&lt;/p>&lt;/li>
&lt;li>&lt;p>Similarly, override &lt;code>autoFocusUnavailable()&lt;/code> in your &lt;code>CameraHost&lt;/code> and, if face
detection is enabled, call &lt;code>stopFaceDetection()&lt;/code> on your &lt;code>CameraFragment&lt;/code> or &lt;code>CameraView&lt;/code>.
&lt;strong>NOTE&lt;/strong>: a dedicated callback for this may be added in the future.&lt;/p>&lt;/li>
&lt;/ol>&lt;p>Note that this capability was added to version 0.5.1 of this library. Also note
that, while you can safely call &lt;code>startFaceDetection()&lt;/code> and &lt;code>stopFaceDetection()&lt;/code>
regardless of API level, &lt;code>getMaxNumDetectedFaces()&lt;/code> should only be called on API Level
14+ devices, or you will be hit with a &lt;code>VerifyError&lt;/code> or the equivalent.&lt;/p>

&lt;h3>&lt;a id="user-content-choosing-a-deviceprofile" class="anchor" href="#choosing-a-deviceprofile" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Choosing a DeviceProfile&lt;/h3>

&lt;p>&lt;code>CameraHost&lt;/code> exists to provide a hook for you to determine how your app
should handle taking pictures and videos. &lt;code>DeviceProfile&lt;/code>, on the other hand,
provides information about how the &lt;em>device&lt;/em> handles taking pictures and videos.
Different devices do slightly different things when working with the camera.
Sometimes this is based on API level, sometimes it is based on how the device
manufacturer tinkered with Android, and sometimes it is based on the underlying
camera hardware. &lt;code>DeviceProfile&lt;/code> provides a place for the CWAC-Camera project
to isolate these differences.&lt;/p>

&lt;p>&lt;code>CameraHost&lt;/code> has a &lt;code>getDeviceProfile()&lt;/code> method that should return an instance
of the &lt;code>DeviceProfile&lt;/code> to use for the device that is running the app.
The implementation of &lt;code>getDeviceProfile()&lt;/code> on &lt;code>SimpleCameraHost&lt;/code>
calls the static &lt;code>getInstance()&lt;/code> method on &lt;code>DeviceProfile&lt;/code>, which chooses
a &lt;code>DeviceProfile&lt;/code> based on internal heuristics. If you encounter problems
with certain devices, you can detect those in your &lt;code>getDeviceProfile()&lt;/code> method
and return a &lt;code>DeviceProfile&lt;/code> that addresses your needs, otherwise settling for
using the library's own choice of &lt;code>DeviceProfile&lt;/code>.&lt;/p>

&lt;p>Note that &lt;code>SimpleCameraHost.Builder&lt;/code> also has a &lt;code>deviceProfile()&lt;/code> setter method that
you can call, passing in a &lt;code>DeviceProfile&lt;/code> that will be used as the default,
replacing the system default.&lt;/p>

&lt;p>The stock &lt;code>DeviceProfile&lt;/code> is largely driven by XML resources. These
resources' names are of the form &lt;code>cwac_camera_profile_XXX_YYY&lt;/code>,
where &lt;code>XXX&lt;/code> is the &lt;code>Build.MANUFACTURER&lt;/code> and &lt;code>YYY&lt;/code> is the &lt;code>Build.MODEL&lt;/code>.
Both &lt;code>Build.MANUFACTURER&lt;/code> and &lt;code>BUILD.MODEL&lt;/code> are converted to lowercase
and have non-alphanumeric values converted to underscores, to ensure
that we wind up with a valid resource filename. Each of those
XML resource files has a &lt;code>&amp;lt;deviceProfile&amp;gt;&lt;/code> root element, containing
child elements for different values that can be overridden:&lt;/p>

&lt;ul>&lt;li>&lt;p>&lt;code>&amp;lt;doesZoomActuallyWork&amp;gt;&lt;/code> (a boolean, &lt;code>true&lt;/code> or &lt;code>false&lt;/code>) overrides
the default zoom detection logic, with &lt;code>false&lt;/code> meaning that the device
lies and zoom is not really supported&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>&amp;lt;maxPictureHeight&amp;gt;&lt;/code> (an integer) is the largest number of pixels
high to use for the camera picture; a &lt;code>Camera.Size&lt;/code> taller than this
is ignored&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>&amp;lt;minPictureHeight&amp;gt;&lt;/code> (an integer) is the smallest number of pixels
high to use for the camera picture; a &lt;code>Camera.Size&lt;/code> shorter than this
is ignored&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>&amp;lt;pictureDelay&amp;gt;&lt;/code> (an integer) is a time in milliseconds to delay
taking the picture after updating &lt;code>Camera.Parameters&lt;/code> with the picture
settings, for devices that seem to reset themselves when the parameters
are updated, resulting in messed-up pictures&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>&amp;lt;portraitFFCFlipped&amp;gt;&lt;/code> (a boolean, &lt;code>true&lt;/code> or &lt;code>false&lt;/code>) indicates if
the image cleanup work needs to flip the image if it was taken in
portrait mode from the front-facing camera&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>&amp;lt;useDeviceOrientation&amp;gt;&lt;/code> (a boolean, &lt;code>true&lt;/code> or &lt;code>false&lt;/code>) indicates
if we should skip the &lt;code>setRotation()&lt;/code> call on &lt;code>Camera.Parameters&lt;/code>
due to a device bug, and instead should just use physical orientation
in the image cleanup phase to get the picture to turn out right&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>&amp;lt;useTextureView&amp;gt;&lt;/code> (a boolean, &lt;code>true&lt;/code> or &lt;code>false&lt;/code>) overrides the
default choice of whether to use a &lt;code>SurfaceView&lt;/code> or a &lt;code>TextureView&lt;/code>
for the preview, normally driven by API level&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;code>&amp;lt;recordingHint&amp;gt;&lt;/code> (&lt;code>ANY&lt;/code>, &lt;code>STILL_ONLY&lt;/code>, &lt;code>VIDEO_ONLY&lt;/code>) indicates whether
we should use a specific type of preview and image capture, based upon
the intended use of the &lt;code>Camera&lt;/code> (this exists solely to work around
some device bugs)&lt;/p>&lt;/li>
&lt;/ul>&lt;p>So long as the resource exists with the right filename, the
library should pick it up, so you can add ones in your app if needed.&lt;/p>

&lt;h3>&lt;a id="user-content-working-directly-with-cameraview" class="anchor" href="#working-directly-with-cameraview" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Working Directly with CameraView&lt;/h3>

&lt;p>If you wish to eschew fragments, you are welcome to work with &lt;code>CameraView&lt;/code>
directly. To do this:&lt;/p>

&lt;ul>&lt;li>&lt;p>Add it in Java code by calling its one-parameter constructor, taking
your &lt;code&gt;Activity&lt;/code> as a parameter. At the present time, &lt;code>CameraView&lt;/code> does not
support being placed in a layout resource.&lt;/p>&lt;/li>
&lt;li>&lt;p>Call &lt;code>setHost()&lt;/code> on the &lt;code>CameraView&lt;/code> as early as possible, to make sure
that the &lt;code>CameraView&lt;/code> is working with the right &lt;code>CameraHost&lt;/code> implementation.
Alternatively, override &lt;code>getHost()&lt;/code> and return the right &lt;code>CameraHost&lt;/code>
there.&lt;/p>&lt;/li>
&lt;li>&lt;p>Forward the &lt;code>onResume()&lt;/code> and &lt;code>onPause()&lt;/code> lifecycle events from your
activity or fragment to the &lt;code>CameraView&lt;/code>.&lt;/p>&lt;/li>
&lt;/ul>&lt;p>Otherwise, &lt;code>CameraView&lt;/code> should work as a regular &lt;code>View&lt;/code>... so long as you do
not try to use it in a layout resource.&lt;/p>

&lt;h3>&lt;a id="user-content-using-cameraview-in-a-layout-resource" class="anchor" href="#using-cameraview-in-a-layout-resource" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Using CameraView in a Layout Resource&lt;/h3>

&lt;p>If you want to use &lt;code>CameraView&lt;/code> in a layout resource, you can, but your
activity will need to implement the &lt;code>CameraHostProvider&lt;/code> interface. This has
one required method: &lt;code>getCameraHost()&lt;/code>, which returns the &lt;code>CameraHost&lt;/code> instance
to be used with the &lt;code>CameraView&lt;/code>. You would implement this in lieu of calling
&lt;code>setHost()&lt;/code> yourself.&lt;/p>

&lt;p>If you want to take advantage of this and use your own layout in a &lt;code>CameraFragment&lt;/code>
subclass, simply override &lt;code>onCreateView()&lt;/code> and do what you want. The only
requirement, other than the &lt;code>CameraHostProvider&lt;/code> mentioned above, is that
your &lt;code>onCreateView()&lt;/code> needs to call the &lt;code>setCameraView()&lt;/code> method, supplying
the &lt;code>CameraView&lt;/code> instance to the superclass.&lt;/p>

&lt;p>The &lt;code>demo-layout/&lt;/code> directory contains a small sample project that demonstrates
this technique.&lt;/p>

&lt;h3>&lt;a id="user-content-flash-modes" class="anchor" href="#flash-modes" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Flash Modes&lt;/h3>

&lt;p>&lt;code>CameraView&lt;/code>, as well as &lt;code>CameraFragment&lt;/code>, has a &lt;code>getFlashMode()&lt;/code> which
returns the flash mode from &lt;code>Camera.Parameters&lt;/code>.&lt;/p>

&lt;p>To adjust the flash mode, you can call &lt;code>flashMode()&lt;/code> on your &lt;code>PictureTransaction&lt;/code>
to specify a mode to apply when the picture is taken. Or, call &lt;code>setFlashMode()&lt;/code>
on &lt;code>CameraView&lt;/code> or &lt;code>CameraFragment&lt;/code> when needed. Or,
you can manually configure the &lt;code>Camera.Parameters&lt;/code> object in &lt;code>adjustPictureParameters()&lt;/code>
and/or &lt;code>adjustPreviewParameters()&lt;/code>.&lt;/p>

&lt;p>The &lt;code>CameraUtils&lt;/code> class has a &lt;code>findBestFlashModeMatch()&lt;/code> method that
takes a &lt;code>Camera.Parameters&lt;/code> object, plus one or more &lt;code>String&lt;/code> names
of flash modes (e.g., &lt;code>Camera.Parameters.FLASH_MODE_ON&lt;/code>), and returns
the mode that appears first in your list of strings that is supported
by the current camera.&lt;/p>

&lt;h2>&lt;a id="user-content-third-party-code" class="anchor" href="#third-party-code" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Third-Party Code&lt;/h2>

&lt;p>kenyee has
&lt;a href="https://github.com/kenyee/cwac-camera/blob/master/library/src/com/commonsware/cwac/camera/asl/CameraFragment.java">an implementation of &lt;code>CameraFragment&lt;/code>&lt;/a>
that simply extends from the
Android Support package's backport of fragments, for use with the AppCompat
backport of the action bar.&lt;/p>

&lt;h2>&lt;a id="user-content-known-limitations" class="anchor" href="#known-limitations" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Known Limitations&lt;/h2>

&lt;p>These are above and beyond &lt;a href="https://github.com/commonsguy/cwac-camera/issues">the bugs filed for this project&lt;/a>:&lt;/p>

&lt;ol>&lt;li>&lt;p>Taking videos in portrait mode is not supported.&lt;/p>&lt;/li>
&lt;li>&lt;p>While a picture or video is being taken, on some devices, the aspect
ratio of the preview gets messed up. The aspect ratio is corrected by &lt;code>CWAC-Camera&lt;/code>
once the picture or video is completed, but more work is needed to try to prevent
this in the first place, or at least mask it a bit better for photos.&lt;/p>&lt;/li>
&lt;li>&lt;p>The Samsung Galaxy Ace refuses to honor a portrait preview in an activity
that itself supports portrait or landscape. If you lock your activity to only
display in landscape, the Galaxy Ace will probably work.&lt;/p>&lt;/li>
&lt;/ol>&lt;h2>&lt;a id="user-content-upgrading" class="anchor" href="#upgrading" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Upgrading&lt;/h2>

&lt;p>If you are moving from an older to a newer edition of CWAC-Camera, here are some
upgrade notes which may help.&lt;/p>

&lt;h3>&lt;a id="user-content-from-05x-to-060-and-higher" class="anchor" href="#from-05x-to-060-and-higher" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>From 0.5.x to 0.6.0 and Higher&lt;/h3>

&lt;p>If you implemented &lt;code>CameraHost&lt;/code> or extended &lt;code>SimpleCameraHost&lt;/code>, note that four
callback methods now receive a &lt;code>PictureTransaction&lt;/code> as the first parameter:&lt;/p>

&lt;ul>&lt;li>&lt;code>adjustPictureParameters(PictureTransaction xact, Camera.Parameters parameters)&lt;/code>&lt;/li>
&lt;li>&lt;code>getPictureSize(PictureTransaction xact, Camera.Parameters parameters)&lt;/code>&lt;/li>
&lt;li>&lt;code>saveImage(PictureTransaction xact, Bitmap bitmap)&lt;/code>&lt;/li>
&lt;li>&lt;code>saveImage(PictureTransaction xact, byte[] image)&lt;/code>&lt;/li>
&lt;/ul>&lt;p>Conversely, all of the old EXIF-related behaviors are now gone. If
you created custom &lt;code>DeviceProfiles&lt;/code>, they will need to be adjusted
to remove the &lt;code>encodesRotationToExif()&lt;/code> and &lt;code>rotateBasedOnExif()&lt;/code>
methods. Also, any custom &lt;code>CameraHost&lt;/code> implementation will need to
have its &lt;code>rotateBasedOnExif()&lt;/code> method removed.&lt;/p>

&lt;p>If you implemented your own &lt;code>DeviceProfile&lt;/code> classes, note that they might
now be able to be replaced by XML resources instead.&lt;/p>

&lt;p>Taking pictures now involves a &lt;code>PictureTransaction&lt;/code>, though the existing
&lt;code>takePicture()&lt;/code&gt; methods are still supported.&lt;/p>

&lt;p>Also note that full-bleed previews are now the default, though you can
override that to revert back to the original behavior if desired.&lt;/p>

&lt;h3>&lt;a id="user-content-from-04x-to-050-and-higher" class="anchor" href="#from-04x-to-050-and-higher" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>From 0.4.x to 0.5.0 and Higher&lt;/h3>

&lt;p>&lt;code>CameraHost&lt;/code> used to have &lt;code>mayUseForVideo()&lt;/code>, returning a boolean. That is
now &lt;code>getRecordingHint()&lt;/code>, returning a &lt;code>CameraHost.RecordingHint&lt;/code> value: &lt;code>STILL_ONLY&lt;/code>,
&lt;code>VIDEO_ONLY&lt;/code>, or &lt;code>ANY&lt;/code>. &lt;code>SimpleCameraHost&lt;/code> was modified to return &lt;code>ANY&lt;/code>, so the
default behavior should be the same as before. Hence, you should only need to worry
about this if you overrode &lt;code>mayUseForVideo()&lt;/code> or implemented your own &lt;code>CameraHost&lt;/code>.&lt;/p>

&lt;p>&lt;code>CameraHost&lt;/code> now has an &lt;code>onCameraFail()&lt;/code> method that takes a &lt;code>FailureReason&lt;/code>
parameter. &lt;code>FailureReason&lt;/code> is an &lt;code>enum&lt;/code>, with values of &lt;code>NO_CAMERAS_REPORTED&lt;/code>
and &lt;code>UNKNOWN&lt;/code> at present. This will be called if &lt;code>CameraView&lt;/code> could not access
a camera. &lt;code>SimpleCameraHost&lt;/code> has an implementation of &lt;code>onCameraFail()&lt;/code> that
just logs a message to LogCat, but you are encouraged to supply your own
implementation that does something more.&lt;/p>

&lt;h3>&lt;a id="user-content-from-02x030-to-040-and-higher" class="anchor" href="#from-02x030-to-040-and-higher" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>From 0.2.x/0.3.0 to 0.4.0 and Higher&lt;/h3>

&lt;p>&lt;code>CameraHost&lt;/code> now requires implementers supply &lt;code>mayUseForVideo()&lt;/code> (&lt;code>true&lt;/code> if the
preview should be optimized for possible use in video recording) and
&lt;code>getPreferredPreviewSizeForVideo()&lt;/code> (returns the preview size to use in case
&lt;code>mayUseForVideo()&lt;/code> returns &lt;code>true&lt;/code>). &lt;code>SimpleCameraHost&lt;/code> provides stock implementations
of these, but if you created your own &lt;code>CameraHost&lt;/code> from scratch, you will need
to add your own versions of these methods.&lt;/p>

&lt;h3>&lt;a id="user-content-from-01x-to-020-and-higher" class="anchor" href="#from-01x-to-020-and-higher" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>From 0.1.x to 0.2.0 and Higher&lt;/h3>

&lt;p>&lt;code>CameraHost&lt;/code> now extends &lt;code>Camera.AutoFocusCallback&lt;/code>, requiring an implementation
of &lt;code>onAutoFocus()&lt;/code>. &lt;code>SimpleCameraHost&lt;/code> shows a basic implementation that, on
API Level 16+, plays the device-standard "hey! you're focused now!" sound.&lt;/p>

&lt;h3>&lt;a id="user-content-from-00x-to-010-and-higher" class="anchor" href="#from-00x-to-010-and-higher" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>From 0.0.x to 0.1.0 and Higher&lt;/h3>

&lt;p>Developers moving from v0.0.x to v0.1.x should note that you now need to pass
a &lt;code>Context&lt;/code> into the constructor of &lt;code>SimpleCameraHost&lt;/code>. This can be any &lt;code>Context&lt;/code>,
as &lt;code>SimpleCameraHost&lt;/code> retrieves the &lt;code>Application&lt;/code> singleton from it, so you do not
have to worry about memory leaks.&lt;/p>

&lt;h3>&lt;a id="user-content-location-based-emergency-service" class="anchor" href="#location-based-emergency-service" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Location based emergency service&lt;/h3>

&lt;h3>&lt;a id="user-content-pi-security-camera" class="anchor" href="#pi-security-camera" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Pi Security Camera&lt;/h3>

&lt;p>Technologies: Raspberry Pi, Firebase/MongoDB, AngularJS&lt;/p>

&lt;p>Given that the state of consumer level security cameras is so bad right now, I think creating a cheap and somewhat robust solution could be a good use of our time. This hack would require a raspberry pi (with wifi) and a cheap usb webcam. The main focus of this hack will be to process incoming images from the camera and detect motion when the camera is armed. It will be challenging to do this on a raspberry pi, given the constraints on processor power. We can then grow this to output a video stream online, and then if possible, also include apps for iOS and Android.&lt;/p>

&lt;h3>&lt;a id="user-content-el-jacket" class="anchor" href="#el-jacket" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>EL Jacket&lt;/h3>

&lt;p>Technologies: Arduino, Raspberry Pi, Spotify, SoundCloud, Twilio, Android, iOS, Firebase/MongoDB, JavaScript&lt;/p>

&lt;p>Take a jacket or something, wire it up with lots of EL wire in different colors, and add music. Have a RPi (or Edison or some other powerful embedded solution) decoding a music source (YouTube, Spotify, SoundCloud, locally stored music - think Beats by ACM), doing a FFT on it to perform frequency analysis, and flash the EL wire according to the different frequencies like a VU meter. Music will be played back on a sufficiently loud speaker connected to the RPi, and people can vote on the next track to play via web interface or by sending a text to a Twilio-powered backend. Network connectivity provided by Wi-Fi or tethered smartphone.&lt;/p>

&lt;p>This idea is currently reserved for Wildhacks&lt;/p>

&lt;h3>&lt;a id="user-content-l4d2-virtual-reality-system" class="anchor" href="#l4d2-virtual-reality-system" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>L4D2 Virtual Reality System&lt;/h3>

&lt;p>Technologies: Arduino, Intel Edison, Raspberry Pi, SourceMod (SourcePawn), Myo, Oculus Rift, Wii remotes, Wiigee-C++, flex sensors&lt;/p>

&lt;p>Take a Nerf gun, rewire its electronics, and connect it to a microcontroller for trigger/reload control. Dual Myos for accelerometer/gyroscope data to determine arm position, angle, and movement for gestures like melee shove, ADS (aim down sights), and throwing objects. Use flex sensors on left hand for additional precision - point to vocalize 'look!', reach out and grab to interact with items, etc. Possible ammo/health counter. Walking can be emulated with 2 Wiimotes strapped to legs, a dance pad, or some other method. Use keybinds to translate from gestures to inputs. Haptic feedback vest with 48 PWM-drivable vibration motors controlled via RPi and LED drivers for feedback when player gets hit, with RGB LED lighting for health visualization (ie. lower brightness as user takes damage, flash all red when player goes down, etc.) To retrieve game data like ammo/health/hit position, write a SourceMod plugin that listens to events and sends data to a client (in this case, the RPi) via sockets which will process the data. Aiming system still unknown, but you can strap a Wiimote to the gun barrel and use a sensor bar to position track the gun.&lt;/p>

&lt;p>Idea currently reserved for a 'big' hackathon (MHacks V), haptic feedback vest might be done separately and incorporated into the overall idea later&lt;/p>

&lt;p>Technologies: Twilio, Android, iOS, Raspberry Pi&lt;/p>

&lt;p>&lt;strong>(Won Best Use of Twilio API at Boilermake) &lt;a href="http://challengepost.com/software/cellularlock">http://challengepost.com/software/cellularlock&lt;/a>&lt;/strong>&lt;/p>

&lt;p>Using a cellular network as the backend of an app. This serves as a proof of concept for building connected digital systems without the need of a reliable internet connection. This is useful mainly becuase of the fact that many developing nations do not have extensive data networks, but do have extensive enough and reliable enough cellular networks to facilitate basic communicatio. This Cellular Backend can be used to help increase the utility of these cellular networks.&lt;/p>

&lt;h1>&lt;a id="user-content-installation" class="anchor" href="#installation" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Installation&lt;/h1>

&lt;p>If you're familliar with &lt;a href="https://www.docker.com/">Docker&lt;/a>, you might want to try using compilebot's &lt;a href="https://registry.hub.docker.com/u/renfredxh/compilebot/">Docker repository&lt;/a>.&lt;/p>

&lt;p>To install compilebot locally, you can follow the following steps. Using &lt;a href="http://www.virtualenv.org">virtualenv&lt;/a> is recommended.&lt;/p>

&lt;div class="highlight highlight-bash">&lt;pre>git clone https://github.com/renfredxh/compilebot.git
&lt;span class="pl-c1">cd&lt;/span> compilebot&lt;/pre>&lt;/div>

&lt;p>You'll need to initialize the ideone-api submodule and install it:&lt;/p>

&lt;div class="highlight highlight-bash">&lt;pre>git submodule init
git submodule update
&lt;span class="pl-c1">cd&lt;/span> lib/ideone-api/
python setup.py install&lt;/pre>&lt;/div>

&lt;p>Install the dependencies:&lt;/p>

&lt;div class="highlight highlight-bash">&lt;pre>pip install -r requirements.txt&lt;/pre>&lt;/div>

&lt;p>Now that all of the dependencies for compilebot have been installed, you'll have to configure the settings file. You can copy the included sample settings into a new file. From the main directory:&lt;/p>

&lt;div class="highlight highlight-bash">&lt;pre>&lt;span class="pl-c1">cd&lt;/span> compilebot
cp sample-config.yml config.yml&lt;/pre>&lt;/div>

&lt;p>Now, if you open &lt;code>config.yml&lt;/code> in your favorite text editor you'll see customizable data for compilebot to use. If you would like to run all of compilebot's features, you'll need a &lt;a href="http://www.reddit.com/">reddit&lt;/a> account and a &lt;a href="http://ideone.com/account/register">ideone&lt;/a> account. Once you have your login credentials, you can edit &lt;code>config.yml&lt;/code> and fill in the following fields:&lt;/p>

&lt;div class="highlight highlight-yaml">&lt;pre>  &lt;span class="pl-s">&lt;span class="pl-ent">reddit_user:&lt;/span> &lt;/span>&lt;span class="pl-c"># Your reddit user name&lt;/span>
  &lt;span class="pl-s">&lt;span class="pl-ent">reddit_pass:&lt;/span> &lt;/span>&lt;span class="pl-c"># Your reddit password&lt;/span>
  &lt;span class="pl-s">&lt;span class="pl-ent">ideone_user:&lt;/span> &lt;/span>&lt;span class="pl-c"># Your ideone username&lt;/span>
  &lt;span class="pl-s">&lt;span class="pl-ent">ideone_pass:&lt;/span> &lt;/span>&lt;span class="pl-c"># Your ideone password&lt;/span>&lt;/pre>&lt;/div>

&lt;p>Finally, try running compilebot:&lt;/p>

&lt;div class="highlight highlight-bash">&lt;pre>python compilebot.py&lt;/pre>&lt;/div>

&lt;p>Install &lt;code>requests&lt;/code> and &lt;code>ideone&lt;/code> using pip or your package manager. (I highly recommend creating a new virtualenv for this.)&lt;br>
You will also need to install this &lt;a href="https://github.com/wchill/piazza-api">Piazza library&lt;/a>; run &lt;code>python setup.py install&lt;/code>.&lt;br>
Make a copy of &lt;code>sample_config.yml&lt;/code> and name it &lt;code>config.yml&lt;/code>. Edit it with your Piazza username/password and Ideone username/API password.&lt;/p>

&lt;h3>&lt;a id="user-content-using-the-bot" class="anchor" href="#using-the-bot" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Using the bot&lt;/h3>

&lt;p>The bot will automatically watch any classes it is a member of and check for new/updated posts, submitting the code in the post to Ideone for compilation if any is found.&lt;/p>

&lt;p>To call the bot, use the following format:&lt;/p>

&lt;blockquote>
&lt;p>CompileBot! python  &lt;/p>

&lt;pre>&lt;code>for i in range(1, 101):
   print i
&lt;/code>&lt;/pre>

&lt;p>Input:&lt;/p>

&lt;pre>&lt;code>blah
&lt;/code>&lt;/pre>
&lt;/blockquote>

&lt;p>CompileBot is case-insensitive and any/no punctuation after its name will be ignored. Code and input should be wrapped in code blocks on Piazza. Input is optional and if none is needed the block can be omitted.&lt;/p>

&lt;h2>&lt;a id="user-content-development-environment-set-up" class="anchor" href="#development-environment-set-up" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Development environment set up&lt;/h2>

&lt;ol>&lt;li>&lt;p>&lt;a href="https://help.github.com/articles/set-up-git/">Set up Git&lt;/a> and install &lt;a href="http://nodejs.org/">node.js&lt;/a>. Node's package manager (&lt;a href="https://www.npmjs.org/">npm&lt;/a>) comes bundled.&lt;/p>&lt;/li>
&lt;li>&lt;p>Globally install &lt;a href="http://bower.io/">Bower&lt;/a> and &lt;a href="http://gruntjs.com/">Grunt&lt;/a>.
You might need to use &lt;code>sudo&lt;/code> or run the command as an Administrator if it fails due to missing permissions,
because of the &lt;a href="https://www.npmjs.org/doc/files/npm-folders.html">location npm installs global packages&lt;/a> in.&lt;br>&lt;code>npm install -g bower grunt-cli&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;a href="https://help.github.com/articles/fork-a-repo/">Fork&lt;/a> this repository. Clone your forked git repository.&lt;br>&lt;code>git clone https://github.com/YOUR-USERNAME/sysbuild.git&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Change to the project directory.&lt;br>&lt;code>cd sysbuild/&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Configure Git to sync your fork with the original repository.&lt;br>&lt;code>git remote add upstream https://github.com/cs-education/sysbuild.git&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Install dependencies and set up the project.&lt;br>&lt;code>npm install&lt;/code>&lt;/p>&lt;/li>
&lt;/ol>&lt;h2>&lt;a id="user-content-useful-commands" class="anchor" href="#useful-commands" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Useful commands&lt;/h2>

&lt;ul>&lt;li>&lt;p>Run a development server. Automatically launches default browser. Also supports live reloading, which means
changes automatically show up without the need to refresh the page. Files are also watched for changes - 
JSHint is automatically run on the changed JS files, changed SASS files are automatically compiled, etc.&lt;br>&lt;code>grunt serve&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>If you add a new bower component, you might want to automatically inject &lt;em>supported&lt;/em> Bower components into the HTML file.&lt;br>&lt;code>grunt bowerInstall&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Run &lt;a href="http://www.jshint.com/about/">JSHint&lt;/a>.&lt;br>&lt;code>grunt jshint&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Run &lt;a href="http://jscs.info/">JSCS&lt;/a>.&lt;br>&lt;code>grunt jscs&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Run tests.&lt;br>&lt;code>grunt test&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Run a local test server, to run the tests in a browser. Navigate to &lt;code>http://localhost:9001&lt;/code> after running the following.&lt;br>&lt;code>grunt testserver&lt;/code>&lt;/p>&lt;/li>
&lt;/ul>&lt;h2>&lt;a id="user-content-contributing" class="anchor" href="#contributing" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Contributing&lt;/h2>

&lt;ol>&lt;li>&lt;p>Make sure your fork is up to date with the upstream repository. See &lt;a href="https://help.github.com/articles/syncing-a-fork/">https://help.github.com/articles/syncing-a-fork/&lt;/a>.&lt;br>&lt;code>git fetch upstream&lt;/code>&lt;br>&lt;code>git checkout master&lt;/code>&lt;br>&lt;code>git merge upstream/master&lt;/code>&lt;br>&lt;code>git push origin master&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Create a new branch to make changes or add a new feature.&lt;br>&lt;code>git checkout -b my_awesome_feature_branch&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Stage your changes before committing. Type the following for every added/modified/deleted file.&lt;br>&lt;code>git add path/to/modified_file&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Commit your changes. Do this often.&lt;br>&lt;code>git commit -m "I changed this to that and fixed bla."&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>Push the branch to origin so that others can see it.&lt;br>&lt;code>git push origin my_awesome_feature_branch&lt;/code>&lt;/p>&lt;/li>
&lt;li>&lt;p>&lt;a href="https://help.github.com/articles/creating-a-pull-request">Create a pull request&lt;/a> for merging into the upstream &lt;code>master&lt;/code>.
Wait for someone to review your code and merge your changes. Make sure you followed the code guidelines below.
If you were working on an issue, you can have the issue &lt;a href="https://github.com/blog/1506-closing-issues-via-pull-requests">closed automatically&lt;/a> when the pull request is merged.&lt;/p>&lt;/li>
&lt;/ol>&lt;h2>&lt;a id="user-content-code-guidelines" class="anchor" href="#code-guidelines" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Code guidelines&lt;/h2>

&lt;ul>&lt;li>&lt;p>We have an &lt;a href="https://github.com/cs-education/sysbuild/blob/master/.editorconfig">editor config&lt;/a> file for maintaining a consistent coding style.
Read more and download plugins at &lt;a href="http://editorconfig.org">http://editorconfig.org&lt;/a>.&lt;/p>&lt;/li>
&lt;li>&lt;p>Please include tests whenever possible.&lt;/p>&lt;/li>
&lt;li>&lt;p>Keep accessibility in mind when writing HTML.&lt;/p>&lt;/li>
&lt;li>&lt;p>Make sure there are no JSHint/JSCS errors.&lt;/p>&lt;/li>
&lt;li>&lt;p>Make sure all tests pass.&lt;/p>&lt;/li>
&lt;li>&lt;p>Avoid pushing changes to &lt;code>master&lt;/code>. Most changes should be in their own branch, which should then be merged into &lt;code>upstream/master&lt;/code> through a pull request.
Your fork's &lt;code>master&lt;/code> should always be in sync with &lt;code>upstream/master&lt;/code>. If you made your changes in your &lt;code>master&lt;/code> branch and your pull request gets rejected,
your &lt;code>master&lt;/code> branch will be ahead of &lt;code>upstream/master&lt;/code> and it is &lt;a href="http://stackoverflow.com/questions/5916329/cleanup-git-master-branch-and-move-some-commit-to-new-branch">hard to cleanup&lt;/a>.
Therefore, always make changes in a new branch.&lt;/p>&lt;/li>
&lt;li>&lt;p>We use &lt;a href="https://travis-ci.org/">Travis CI&lt;/a> for continuous integration. Every time you open a pull request and make commits onto it, a build is triggered.
Sometimes you will make commits which do not need a build to be created (for example, editing the README or non-code changes). In that case, just add
&lt;code>[skip ci]&lt;/code> somewhere in your commit message. &lt;a href="http://docs.travis-ci.com/user/how-to-skip-a-build/">Learn more&lt;/a>.&lt;/p>&lt;/li>
&lt;/ul>&lt;h2>&lt;a id="user-content-deploying" class="anchor" href="#deploying" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Deploying&lt;/h2>

&lt;p>We use &lt;a href="https://help.github.com/articles/what-are-github-pages">GitHub Pages&lt;/a> for hosting the application.
The production repository is &lt;a href="https://github.com/cs-education/sys">https://github.com/cs-education/sys&lt;/a> and the staging repository is &lt;a href="https://github.com/cs-education/sys-staging">https://github.com/cs-education/sys-staging&lt;/a>.
You will need push access to the appropriate repository before you can deploy.&lt;/p>

&lt;ol>&lt;li>&lt;p>Build the distributable project (output in the &lt;code>dist/&lt;/code> folder).&lt;br>&lt;code>grunt build&lt;/code>&lt;/p>&lt;/li>

&lt;p>A JavaScript library for arbitrary-precision decimal and non-decimal arithmetic.&lt;/p>

&lt;p>&lt;a href="https://travis-ci.org/MikeMcl/bignumber.js">&lt;img src="https://camo.githubusercontent.com/836112d3840468731e8daaf085882bf1ba382230/68747470733a2f2f7472617669732d63692e6f72672f4d696b654d636c2f6269676e756d6265722e6a732e737667" alt="Build Status" data-canonical-src="https://travis-ci.org/MikeMcl/bignumber.js.svg" style="max-width:100%;">&lt;/a>&lt;/p>

&lt;p>&lt;br>&lt;/p>

&lt;h2>&lt;a id="user-content-features" class="anchor" href="#features" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Features&lt;/h2>

&lt;ul>&lt;li>Faster, smaller, and perhaps easier to use than JavaScript versions of Java's BigDecimal&lt;/li>
&lt;li>8 KB minified and gzipped&lt;/li>
&lt;li>Simple API but full-featured&lt;/li>
&lt;li>Works with numbers with or without fraction digits in bases from 2 to 64 inclusive&lt;/li>
&lt;li>Replicates the &lt;code>toExponential&lt;/code>, &lt;code>toFixed&lt;/code>, &lt;code>toPrecision&lt;/code> and &lt;code>toString&lt;/code> methods of JavaScript's Number type&lt;/li>
&lt;li>Includes a &lt;code>toFraction&lt;/code> and a correctly-rounded &lt;code>squareRoot&lt;/code> method&lt;/li>
&lt;li>Supports cryptographically-secure pseudo-random number generation&lt;/li>
&lt;li>No dependencies&lt;/li>
&lt;li>Wide platform compatibility: uses JavaScript 1.5 (ECMAScript 3) features only&lt;/li>
&lt;li>Comprehensive &lt;a href="http://mikemcl.github.io/bignumber.js/">documentation&lt;/a> and test set&lt;/li>
&lt;/ul>&lt;p>&lt;a href="https://raw.githubusercontent.com/MikeMcl/bignumber.js/gh-pages/API.png" target="_blank">&lt;img src="https://raw.githubusercontent.com/MikeMcl/bignumber.js/gh-pages/API.png" alt="API" style="max-width:100%;">&lt;/a>&lt;/p>

&lt;p>If a smaller and simpler library is required see &lt;a href="https://github.com/MikeMcl/big.js/">big.js&lt;/a>.
It's less than half the size but only works with decimal numbers and only has half the methods.
It also does not allow &lt;code>NaN&lt;/code> or &lt;code>Infinity&lt;/code>, or have the configuration options of this library.&lt;/p>

&lt;p>See also &lt;a href="https://github.com/MikeMcl/decimal.js/">decimal.js&lt;/a>, which among other things adds support for non-integer powers, and performs all operations to a specified number of significant digits.&lt;/p>

&lt;h2>&lt;a id="user-content-load" class="anchor" href="#load" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Load&lt;/h2>

&lt;p>The library is the single JavaScript file &lt;em>bignumber.js&lt;/em> (or minified, &lt;em>bignumber.min.js&lt;/em>).&lt;/p>

&lt;div class="highlight highlight-html">&lt;pre>&amp;lt;&lt;span class="pl-ent">script&lt;/span> &lt;span class="pl-e">src&lt;/span>=&lt;span class="pl-s">&lt;span class="pl-pds">'&lt;/span>relative/path/to/bignumber.js&lt;span class="pl-pds">'&lt;/span>&lt;/span>&amp;gt;&amp;lt;/&lt;span class="pl-ent">script&lt;/span>&amp;gt;&lt;/pre>&lt;/div>

&lt;p>For &lt;a href="http://nodejs.org">Node.js&lt;/a> or &lt;a href="https://iojs.org/en/index.html">io.js&lt;/a>, the library is available from the &lt;a href="https://npmjs.org/">npm&lt;/a> registry&lt;/p>

&lt;pre>&lt;code>$ npm install bignumber.js
&lt;/code>&lt;/pre>

&lt;div class="highlight highlight-javascript">&lt;pre>&lt;span class="pl-k">var&lt;/span> BigNumber &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-c1">require&lt;/span>(&lt;span class="pl-s">&lt;span class="pl-pds">'&lt;/span>bignumber.js&lt;span class="pl-pds">'&lt;/span>&lt;/span>);&lt;/pre>&lt;/div>

&lt;p>To load with AMD loader libraries such as &lt;a href="http://requirejs.org/">requireJS&lt;/a>:&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>&lt;span class="pl-c1">require&lt;/span>([&lt;span class="pl-s">&lt;span class="pl-pds">'&lt;/span>path/to/bignumber&lt;span class="pl-pds">'&lt;/span>&lt;/span>], &lt;span class="pl-k">function&lt;/span>(&lt;span class="pl-smi">BigNumber&lt;/span>) {  
    &lt;span class="pl-c">// Use BigNumber here in local scope. No global BigNumber.&lt;/span>
});&lt;/pre>&lt;/div>

&lt;h2>&lt;a id="user-content-use" class="anchor" href="#use" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Use&lt;/h2>

&lt;p>&lt;em>In all examples below, &lt;code>var&lt;/code>, semicolons and &lt;code>toString&lt;/code> calls are not shown.
If a commented-out value is in quotes it means &lt;code>toString&lt;/code> has been called on the preceding expression.&lt;/em>&lt;/p>

&lt;p>The library exports a single function: &lt;code>BigNumber&lt;/code>, the constructor of BigNumber instances.&lt;/p>

&lt;p>It accepts a value of type number &lt;em>(up to 15 significant digits only)&lt;/em>, string or BigNumber object,&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>x &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">123.4567&lt;/span>)
y &lt;span class="pl-k">=&lt;/span> BigNumber(&lt;span class="pl-s">&lt;span class="pl-pds">'&lt;/span>123456.7e-3&lt;span class="pl-pds">'&lt;/span>&lt;/span>)
z &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(x)
x.equals(y) &lt;span class="pl-k">&amp;amp;&amp;amp;&lt;/span> y.equals(z) &lt;span class="pl-k"&gt;&amp;amp;&amp;amp;&lt;/span> x.equals(z)      &lt;span class="pl-c">// true&lt;/span>&lt;/pre>&lt;/div>

&lt;p>and a base from 2 to 64 inclusive can be specified.&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>x &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">1011&lt;/span>, &lt;span class="pl-c1">2&lt;/span>)          &lt;span class="pl-c">// "11"&lt;/span>
y &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-s">&lt;span class="pl-pds">'&lt;/span>zz.9&lt;span class="pl-pds">'&lt;/span>&lt;/span>, &lt;span class="pl-c1">36&lt;/span>)       &lt;span class="pl-c">// "1295.25"&lt;/span>
z &lt;span class="pl-k">=&lt;/span> x.plus(y)                       &lt;span class="pl-c">// "1306.25"&lt;/span>&lt;/pre>&lt;/div>

&lt;p>A BigNumber is immutable in the sense that it is not changed by its methods.  &lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>&lt;span class="pl-c1">0.3&lt;/span> &lt;span class="pl-k">-&lt;/span> &lt;span class="pl-c1">0.1&lt;/span>                           &lt;span class="pl-c">// 0.19999999999999998  &lt;/span>
x &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">0.3&lt;/span>)
x.minus(&lt;span class="pl-c1">0.1&lt;/span>)                        &lt;span class="pl-c">// "0.2"&lt;/span>
x                                   &lt;span class="pl-c">// "0.3"&lt;/span>&lt;/pre>&lt;/div>

&lt;p>The methods that return a BigNumber can be chained.&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>x.dividedBy(y).plus(z).times(&lt;span class="pl-c1">9&lt;/span>).&lt;span class="pl-c1">floor&lt;/span>()
x.times(&lt;span class="pl-s">&lt;span class="pl-pds">'&lt;/span>1.23456780123456789e+9&lt;span class="pl-pds">'&lt;/span>&lt;/span>).plus(&lt;span class="pl-c1">9876.5432321&lt;/span>).dividedBy(&lt;span class="pl-s">&lt;span class="pl-pds">'&lt;/span>4444562598.111772&lt;span class="pl-pds">'&lt;/span>&lt;/span>).&lt;span class="pl-c1">ceil&lt;/span>()&lt;/pre>&lt;/div>

&lt;p>Many method names have a shorter alias.&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>x.squareRoot().dividedBy(y).toPower(&lt;span class="pl-c1">3&lt;/span>).equals(x.&lt;span class="pl-c1">sqrt&lt;/span>().div(y).&lt;span class="pl-c1">pow&lt;/span>(&lt;span class="pl-c1">3&lt;/span>))         &lt;span class="pl-c">// true&lt;/span>
x.cmp(y.mod(z).neg()) &lt;span class="pl-k">==&lt;/span> &lt;span class="pl-c1">1&lt;/span> &lt;span class="pl-k">&amp;amp;&amp;amp;&lt;/span> x.comparedTo(y.modulo(z).negated()) &lt;span class="pl-k">==&lt;/span> &lt;span class="pl-c1">1&lt;/span>        &lt;span class="pl-c">// true&lt;/span>&lt;/pre>&lt;/div>

&lt;p>Like JavaScript's number type, there are &lt;code>toExponential&lt;/code>, &lt;code>toFixed&lt;/code> and &lt;code>toPrecision&lt;/code> methods&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>x &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">255.5&lt;/span>)
x.toExponential(&lt;span class="pl-c1">5&lt;/span>)                  &lt;span class="pl-c">// "2.55500e+2"&lt;/span>
x.toFixed(&lt;span class="pl-c1">5&lt;/span>)                        &lt;span class="pl-c">// "255.50000"&lt;/span>
x.toPrecision(&lt;span class="pl-c1">5&lt;/span>)                    &lt;span class="pl-c">// "255.50"&lt;/span>
x.toNumber()                        &lt;span class="pl-c">// 255.5&lt;/span>&lt;/pre>&lt;/div>

&lt;p>and a base can be specified for &lt;code>toString&lt;/code>.&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre> x.&lt;span class="pl-c1">toString&lt;/span>(&lt;span class="pl-c1">16&lt;/span>)                     &lt;span class="pl-c">// "ff.8"&lt;/span>&lt;/pre>&lt;/div>

&lt;p>There is also a &lt;code>toFormat&lt;/code> method which may be useful for internationalisation&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>y &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-s">&lt;span class="pl-pds">'&lt;/span>1234567.898765&lt;span class="pl-pds">'&lt;/span>&lt;/span>)
y.toFormat(&lt;span class="pl-c1">2&lt;/span>)                       &lt;span class="pl-c">// "1,234,567.90"&lt;/span>&lt;/pre>&lt;/div>

&lt;p>The maximum number of decimal places of the result of an operation involving division (i.e. a division, square root, base conversion or negative power operation) is set using the &lt;code>config&lt;/code> method of the &lt;code>BigNumber&lt;/code> constructor.&lt;/p>

&lt;p>The other arithmetic operations always give the exact result.&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>BigNumber.config({ DECIMAL_PLACES&lt;span class="pl-k">:&lt;/span> &lt;span class="pl-c1">10&lt;/span>, ROUNDING_MODE&lt;span class="pl-k">:&lt;/span> &lt;span class="pl-c1">4&lt;/span> })
&lt;span class="pl-c">// Alternatively, BigNumber.config( 10, 4 );&lt;/span>

x &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">2&lt;/span>);
y &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">3&lt;/span>);
z &lt;span class="pl-k">=&lt;/span> x.div(y)                        &lt;span class="pl-c">// "0.6666666667"&lt;/span>
z.&lt;span class="pl-c1">sqrt&lt;/span>()                            &lt;span class="pl-c">// "0.8164965809"&lt;/span>
z.&lt;span class="pl-c1">pow&lt;/span>(&lt;span class="pl-k">-&lt;/span>&lt;span class="pl-c1">3&lt;/span>)                           &lt;span class="pl-c">// "3.3749999995"&lt;/span>
z.&lt;span class="pl-c1">toString&lt;/span>(&lt;span class="pl-c1">2&lt;/span>)                       &lt;span class="pl-c">// "0.1010101011"&lt;/span>
z.times(z)                          &lt;span class="pl-c">// "0.44444444448888888889"&lt;/span>
z.times(z).&lt;span class="pl-c1">round&lt;/span>(&lt;span class="pl-c1">10&lt;/span>)                &lt;span class="pl-c">// "0.4444444445"&lt;/span>&lt;/pre>&lt;/div>

&lt;p>There is a &lt;code>toFraction&lt;/code> method with an optional &lt;em>maximum denominator&lt;/em> argument&lt;/p&gt;

&lt;div class="highlight highlight-javascript">&lt;pre>y &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">355&lt;/span>)
pi &lt;span class="pl-k">=&lt;/span> y.dividedBy(&lt;span class="pl-c1">113&lt;/span>)               &lt;span class="pl-c">// "3.1415929204"&lt;/span>
pi.toFraction()                     &lt;span class="pl-c">// [ "7853982301", "2500000000" ]&lt;/span>
pi.toFraction(&lt;span class="pl-c1">1000&lt;/span>)                 &lt;span class="pl-c">// [ "355", "113" ]&lt;/span>&lt;/pre>&lt;/div>

&lt;p>and &lt;code>isNaN&lt;/code> and &lt;code>isFinite&lt;/code> methods, as &lt;code>NaN&lt;/code> and &lt;code>Infinity&lt;/code> are valid &lt;code>BigNumber&lt;/code> values.&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>x &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">NaN&lt;/span>)                                           &lt;span class="pl-c">// "NaN"&lt;/span>
y &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">Infinity&lt;/span>)                                      &lt;span class="pl-c">// "Infinity"&lt;/span>
x.&lt;span class="pl-c1">isNaN&lt;/span>() &lt;span class="pl-k">&amp;amp;&amp;amp;&lt;/span> &lt;span class="pl-k">!&lt;/span>y.&lt;span class="pl-c1">isNaN&lt;/span>() &lt;span class="pl-k">&amp;amp;&amp;amp;&lt;/span> &lt;span class="pl-k">!&lt;/span>x.&lt;span class="pl-c1">isFinite&lt;/span>() &lt;span class="pl-k">&amp;amp;&amp;amp;&lt;/span> &lt;span class="pl-k">!&lt;/span>y.&lt;span class="pl-c1">isFinite&lt;/span>()        &lt;span class="pl-c">// true&lt;/span&gt;&lt;/pre>&lt;/div>

&lt;p>The value of a BigNumber is stored in a decimal floating point format in terms of a coefficient, exponent and sign.&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>x &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-k">-&lt;/span>&lt;span class="pl-c1">123.456&lt;/span>);
x.c                                 &lt;span class="pl-c">// [ 123, 45600000000000 ]  coefficient (i.e. significand)&lt;/span>
x.e                                 &lt;span class="pl-c">// 2                        exponent&lt;/span>
x.s                                 &lt;span class="pl-c">// -1                       sign&lt;/span>&lt;/pre>&lt;/div>

&lt;p>Multiple BigNumber constructors can be created, each with their own independent configuration which applies to all BigNumber's created from it.&lt;/p>

&lt;div class="highlight highlight-javascript">&lt;pre>&lt;span class="pl-c">// Set DECIMAL_PLACES for the original BigNumber constructor&lt;/span>
BigNumber.config({ DECIMAL_PLACES&lt;span class="pl-k">:&lt;/span&gt; &lt;span class="pl-c1">10&lt;/span> })

&lt;span class="pl-c">// Create another BigNumber constructor, optionally passing in a configuration object&lt;/span>
&lt;span class="pl-c1">BN&lt;/span> &lt;span class="pl-k">=&lt;/span> BigNumber.another({ DECIMAL_PLACES&lt;span class="pl-k">:&lt;/span> &lt;span class="pl-c1">5&lt;/span> })

x &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BigNumber&lt;/span>(&lt;span class="pl-c1">1&lt;/span>)
y &lt;span class="pl-k">=&lt;/span> &lt;span class="pl-k">new&lt;/span> &lt;span class="pl-en">BN&lt;/span>(&lt;span class="pl-c1">1&lt;/span>)

x.div(&lt;span class="pl-c1">3&lt;/span>)                            &lt;span class="pl-c">// '0.3333333333'&lt;/span>
y.div(&lt;span class="pl-c1">3&lt;/span>)                            &lt;span class="pl-c">// '0.33333'&lt;/span>&lt;/pre>&lt;/div>

&lt;p>For futher information see the &lt;a href="http://mikemcl.github.io/bignumber.js/">API&lt;/a> reference in the &lt;em>doc&lt;/em> directory.&lt;/p>

&lt;h2>&lt;a id="user-content-test" class="anchor" href="#test" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Test&lt;/h2>

&lt;p>The &lt;em>test&lt;/em> directory contains the test scripts for each method.&lt;/p>

&lt;p>The tests can be run with Node or a browser. For Node use&lt;/p>

&lt;pre>&lt;code>$ npm test
&lt;/code>&lt;/pre>

&lt;p>or&lt;/p>

&lt;pre>&lt;code>$ node test/every-test
&lt;/code>&lt;/pre>

&lt;p>To test a single method, e.g.&lt;/p>

&lt;pre>&lt;code>$ node test/toFraction
&lt;/code>&lt;/pre>

&lt;p>For the browser, see &lt;em>every-test.html&lt;/em> and &lt;em>single-test.html&lt;/em> in the &lt;em>test/browser&lt;/em> directory.  &lt;/p>

&lt;p>&lt;em>bignumber-vs-number.html&lt;/em> enables some of the methods of bignumber.js to be compared with those of JavaScript's number type.  &lt;/p>

&lt;h2>&lt;a id="user-content-versions" class="anchor" href="#versions" aria-hidden="true">&lt;span class="octicon octicon-link">&lt;/span>&lt;/a>Versions&lt;/h2>

&lt;p>This is version 2.x.x of the library, for version 1.x.x see the tagged releases or switch to the 'original' branch. The advantages of version 2 are that it is considerably faster for numbers with many digits and that there are a some added methods (see Change Log below). The disadvantages are more lines of code and increased code complexity, and the loss of simplicity in no longer having the coefficient of a BigNumber stored in base 10. The 'original' version will continue to be supported.&lt;/p>
